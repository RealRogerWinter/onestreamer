## [Unreleased]

### Fixed
- **AI moderation pipeline — LiveKit `transcription-chunk` payload shape (production hotfix to [ADR-0013](docs/architecture/adr/0013-ai-moderation-pipeline.md) chain).** Discovered while verifying the freshly-deployed AI moderation service against the live URL-relay: every chunk was being silently dropped by `ModerationService.handleTranscriptChunk` because of a payload-shape divergence in `TranscriptionService`. The MediaSoup branch of `startTimedTranscription` emits `transcription-chunk` with `{ ..., text }` (`server/services/TranscriptionService.js` around line 1061) but the LiveKit branch — which is the path URL-relay streams actually take — emits `{ ..., transcription }` (around line 997). `handleTranscriptChunk` strictly read `chunk.text`, so the production URL-relay path was unmoderated end-to-end despite the service initializing successfully (the API endpoints, seed data, and event-emitter subscription all reported healthy). Verified empirically against the live module: the same call with `{text}` writes a `moderation_events` row + fires the notifier; the same call with `{transcription}` (and the same offending phrase) returned null with no row. The fix is a defensive read in `handleTranscriptChunk` — `chunk.text || chunk.transcription` — that normalizes both shapes down to a single `text` field before passing the chunk through `_processChunk`, so the rest of the pipeline (sliding-overlap matching, Stage 2 call payload, the `transcript_excerpt` we persist) doesn't need a per-callsite ternary. Tie-breaker: when both fields are present, `text` wins (the MediaSoup-emit shape is canonical; LiveKit is the legacy outlier). The proper structural fix — normalizing the emit shape inside `TranscriptionService` so consumers don't have to read both fields — expands the blast radius (every `transcription-chunk` consumer in the codebase has to be re-audited) and is intentionally left as a tracked follow-up; the defensive read carries the production responsibility in the meantime. **Tests**: 3 new regression tests in `server/tests/services/ModerationService.test.js` pinning the contract: LiveKit-shape `{transcription}` produces the same `admin_review` row and notifier emit as MediaSoup-shape `{text}`; empty `{transcription: ""}` is rejected like empty `{text: ""}`; when both fields are present, `text` is preferred. End-to-end verified via a one-off diagnostic against the actual `ModerationService` module: `{transcription: "this is a faggot test"}` now writes id=1 with `decision=admin_review` and fires `notifier.eventCreated`. 602/602 server tests pass (+3 new).

### Added
- **AI moderation pipeline — retention purge + GDPR Article 15 export (PR-M6 of [ADR-0013](docs/architecture/adr/0013-ai-moderation-pipeline.md)).** Seventh of eight planned PRs landing the AI moderation pipeline. Closes the two compliance essentials the plan calls out as non-negotiable for production: data retention enforcement (so we're not silently growing a 1-million-row hate-speech audit log) and a GDPR Article 15 right-of-access export endpoint. Defers (with explicit notes in the deferred list at the bottom of this entry) the lower-priority M6 items: two-admin approval, hash-chained audit, Prometheus metrics, automatic `AI_MODERATION_ENFORCE=true` flip. **Retention scheduler**: new `ModerationService.startRetentionScheduler({ flaggedRetentionDays = 90, cleanRetentionDays = 30, intervalMs })` schedules a 24-hour `setInterval` that calls `purgeOldEvents()` — deletes `moderation_events` rows older than 90 days where `final_decision <> 'clean'` and rows older than 30 days where `final_decision = 'clean'` (the latter is a placeholder window — PR-M1 doesn't write `clean` rows, only Stage 1 hits — but the constraint is wired so a future PR that starts logging clean classifications gets the right retention without code changes). First tick is delayed 60s after boot so retention doesn't compete with other boot-time IO. Both the first-run timeout and the recurring interval are `.unref()`'d so SIGTERM exits cleanly. `moderationService.stop()` now also clears both timers. Retention windows are env-overridable via `AI_MODERATION_RETENTION_FLAGGED_DAYS` / `AI_MODERATION_RETENTION_CLEAN_DAYS` for the rare case (data-deletion request, legal hold, etc.) where ops needs a non-default window. **GDPR export endpoint**: new `GET /api/moderation-ai/me/export` (note: `authenticateToken`, not `authenticateAdmin` — every authenticated user can pull their own data per GDPR Article 15). Returns `{user_id, generated_at, notice, legal_basis: 'GDPR Article 6(1)(f) — legitimate safety interest', event_count, events: [...]}`. Match logic walks `sessionService.socketToUserId` for socket ids currently mapped to the caller's user id, then pulls all `moderation_events` rows with `streamer_id` in that set. **This is best-effort and explicitly documented as such in the response `notice` field**: the SessionService map reflects CURRENT live sockets only — a user whose moderation event was captured against a now-disconnected socket id will see those events ONLY if they're reconnected on a socket that still maps to the same user id. Persistent socketId → userId history (which would give us full historical recall) is deferred to a future PR; M6's "best-effort + notice" stance is the GDPR-defensible minimum for v1. **Right-to-erasure (GDPR Article 17)** is NOT implemented as a self-service endpoint: some moderation_events retention is "necessary for legitimate safety interest" per GDPR Article 6(1)(f) and cannot be erased on demand, so erasure runs through a separate request-and-review flow (admin-mediated; deferred to follow-up). The response notice spells this out so end users have a clear expectation. **Wiring**: `server/index.js` calls `moderationService.startRetentionScheduler()` right after the existing `moderationService.initialize()` resolves, before chatBotService gets the moderation gate. The route file imports `authenticateToken` alongside the existing `authenticateAdmin`. **Tests**: 4 new ModerationService tests covering flagged-row retention (100-day-old row deleted, fresh row kept), clean-row retention (40 days deleted, 10 days kept), different-window cross-check (50d flagged stays, 50d clean dies), and scheduler idempotency + stop() clears the timer. 3 new route tests covering /me/export returning events for the caller's session-mapped socket ids, returning empty when the caller has no socket history, and returning 400 when the token has no user id. 541/541 server tests pass (+7 new). **Deferred from M6 to a future PR (M6.5 / M7 / M8)**: (1) two-admin approval for moderation_terms mutations and ban reversals — single-admin is acceptable per the M0 "out of scope" note; (2) hash-chained `moderation_terms_audit.prev_hash` / `row_hash` for tamper evidence — rows are written today and useful, the chain itself can be backfilled; (3) Prometheus-shaped metrics — `console.log` lines suffice until ops actually has an observability target; (4) automatic flip of `AI_MODERATION_ENFORCE=true` — that's an operator decision in the env file, not a code change; (5) Article 17 self-service erasure flow; (6) persistent socketId → userId history for full GDPR recall. **PR-M7 (URL-relay numeric-ID hardening)** also remains deferred. The current ActionArbiter already passes `event.external_user_id` to the WhitelistService when one exists on the event row, but neither the schema (`url_relay_filter_entries`) nor the matcher consults that column yet. The next-PR work is a small migration adding `external_user_id` to the schema with an idempotent ALTER, plus extending `WhitelistService._refreshCache` to build a `blockStreamerIds: Set` and `WhitelistService.checkAllowed` to OR-match against it — bounded ~150 lines + tests, naturally a small follow-up.

- **AI moderation pipeline — Whisper hardening + MovieBot output gate (PR-M4 of [ADR-0013](docs/architecture/adr/0013-ai-moderation-pipeline.md)).** Fifth of eight PRs landing the AI moderation pipeline. Closes the two known mid-pipeline holes that PR-M0/M1/M2/M3 left open: (1) Whisper's default behaviour of redacting profanity to `***` (which defeats the Stage 1 word-filter) and (2) the MovieBot output path, where the bot's LLM reply could itself contain offensive content (especially under prompt injection from the transcript). **Whisper hardening**: `TranscriptionService.transcribeWithWhisperCpp` now passes `--temperature 0.0` (deterministic output — re-running produces the same transcript, important for moderation evidence) and `--prompt "Transcribe verbatim, including any profanity."` (steers Whisper away from the asterisk-redaction behaviour documented in the OpenAI Whisper community discussions). The prompt is short and stylistically neutral so it can't fight the user's language or accent. **Full JSON confidence parsing** (`-oj --word-thold 0.6 --no-speech-thold 0.6` + `whisper_avg_logprob` / `whisper_no_speech_prob` columns wired into `transcription-chunk` event payload) and the **two-confirming-chunks rule** for the slur category are deferred — both require either reading whisper.cpp's `<audiofile>.json` sidecar (a second IO operation) or stitching word-level probability data through the transcription emit shape, and are tracked as M4.5 follow-ups. The chunk-level confidence drop is the cleanest defense against the documented Whisper "violent-content hallucination" failure mode; the deferred work is the load-bearing v1.5 hardening. **MovieBot output gate**: new `ModerationService.checkBotOutput(text, ctx)` runs Stage 1 + Stage 2 (no Stage 3, no ActionArbiter — drop is the only action) on every generated bot reply before chat-service relay. Decision rule is slightly stricter than the streamer-audio pipeline: drop on (a) any hard-tier word match OR (b) Stage 1 match + Stage 2 risk_level >= 2. Threshold is lower than the streamer pipeline (which auto-bans only on 2-of-2 + risk_level=3) because bot output is a controlled surface — we'd rather drop a borderline reply than emit it under the platform's identity. Dropped events write `stream_type='moviebot-output'`, `final_decision='mb_output_dropped'`, and one of `action_taken='dropped_hard_tier_word'` / `'dropped_stage2_risk'`; `ModerationNotifier.botOutputDropped()` fans the row to admin sockets. `ChatBotService.generateMovieComment` now takes a `setModerationService()` setter (late-injected from `server/index.js` right after `moderationService.initialize()` resolves) and gates the existing `socket.emit('send-message', ...)` path — flagged replies return `{success: false, error: 'moderation_dropped:<reason>', moderation_event_id}` which MovieBotService's existing `if (response.success && response.message)` guard already treats as a no-op, so no MovieBot-side change is required to suppress the chat emission. **Fail-open on bug**: if the moderation gate throws (rather than returning `{allowed: false}`), ChatBotService swallows the error and lets the message through — a bug in the gate should not silence the bot. The streamer-audio pipeline (Stage 1+2+3 on Whisper transcripts) is the next-tier defense for that case. **Tests**: 6 new tests in `ModerationService.test.js` covering clean-text pass-through, hard-tier-word drop, soft-tier-without-Stage-2-risk pass-through (the reclaimed AAVE / "nigga" case is the load-bearing one — Stage 2 returning risk_level=1 means we DON'T drop), soft-tier + Stage 2 risk=2 drop, empty/null input pass-through, and post-stop short-circuit. 517/517 server tests pass (+6 new).
- **AI moderation pipeline — Stage 3 OpenAI cross-check + ActionArbiter, enforcement behind feature flag (PR-M3 of [ADR-0013](docs/architecture/adr/0013-ai-moderation-pipeline.md)).** Fourth of eight PRs landing the AI moderation pipeline. This is the first PR with a runtime action surface — but it's gated behind `AI_MODERATION_ENFORCE=true` (default false), so a host that ships through M3 sees the full pipeline running in log-only mode with `final_decision='admin_review'` rows until ops flips the flag in M6. New `ModerationStage3.js` is a self-contained client for OpenAI's free `omni-moderation-latest` endpoint — fires ONLY when Stage 2 returns `risk_level=3`, returns the standard `{flagged, categories, scores, model, latency_ms}` shape, has the same circuit-breaker contract as Stage 2 (5 consecutive failures opens a 60s window, single success resets). Unlike Stage 2 it does NOT wrap input in anti-injection delimiters because omni-moderation is a classifier endpoint (not chat-completions) — prompt injection is structurally impossible against a model that doesn't follow instructions. `OPENAI_API_KEY` is optional: when absent, high-risk events route to `admin_review` rather than auto-action, which is the safe default. New `ModerationActionArbiter.js` is a stateless service that takes a fully-classified event and turns it into the appropriate action. **Decision tree**: stale-session check first (if `event.stream_session_id !== streamService.getStreamGeneration()` the offending streamer has already rotated for an unrelated reason — downgrade to `admin_review` so we don't ban whoever is currently on-air); `enforce=false` gate (M3 default — downgrade to `admin_review` even on 2-of-2 HIGH agreement); then branch on `stream_type`: `'webcam'` → `userRepository.banFromStreaming(userId, 'ai-moderation')` if the streamer is authenticated, emit streamer-banner via the notifier (with `/admin/moderation/events/:id` appeal URL), call `randomStreamRotationService._rotateToNewStream()`, final_decision = `auto_ban`; `'url-relay'` → `whitelistService.addEntry({platform, entry_type:'streamer', value: login, list:'block'}, 'ai-moderator')` (login-keyed for M3; PR-M7 schema-bumps to numeric `external_user_id` for spoof resistance), rotation, final_decision = `auto_skip`; `'viewbot'` / `'moviebot-output'` / unknown → log only, no action. The arbiter never throws — every step is wrapped so a failing ban write or rotation produces a structured `action_taken` string rather than corrupting the event row. UNIQUE-constraint hits on the URL-relay blocklist (the streamer is already blocked) are treated as no-op success. **`ModerationService` 2-of-2 agreement gate**: when `stage2.risk_level=3` and Stage 3 is ready and the per-streamer quota allows (default 20 calls/hour, rolling-window enforced), Stage 3 is called with `surroundingContext + currentText`. If Stage 3 flags, the arbiter runs; if Stage 3 doesn't flag the event downgrades to `admin_review` with `action_taken='stage3_disagreed'`; if Stage 3 returns degraded the event becomes `deferred_degraded`. Per-streamer Stage 3 quota uses a per-streamer rolling timestamp list trimmed to the last hour on every check — defeats the "Stage-3 burn" cost attack from the red-team review. Two new `setActionArbiter()` / `setStage3()` setters on ModerationService support the late-injection pattern that `server/index.js` needs: ModerationService is constructed eagerly in the bootstrap section before `RandomStreamRotationService` exists, then both the MediaSoup and the LiveKit branches construct an ActionArbiter (with `enforce: process.env.AI_MODERATION_ENFORCE === 'true'`) once their rotation service is wired and call `moderationService.setActionArbiter()`. The arbiter is also exposed on `app.locals.moderationActionArbiter` for the future admin reversal route (PR-M5). **Tests**: 33 new tests across two new test files plus 7 added to `ModerationService.test.js`. `ModerationStage3.test.js` covers contract + happy path + error paths (non-2xx, malformed envelope, missing results) + circuit-breaker lifecycle, plus an explicit assertion that the body is sent as raw `input` without the anti-injection wrapper (classifier endpoint, structurally injection-immune). `ModerationActionArbiter.test.js` covers all four branches (stale-session, enforce-off, webcam authenticated/anonymous/rotation-fail/ban-fail, url-relay add/UNIQUE-already-blocked/cannot_block, viewbot/unknown). `ModerationService.test.js` adds the integration tests for Stage 3 not-called-when-low-risk, Stage 3 called on risk=3 → agreement → arbiter, disagreement → admin_review, degraded → deferred_degraded, not-ready → admin_review, per-streamer quota gating, and setActionArbiter replacement. 511/511 server tests pass (+33 new).
- **AI moderation pipeline — Stage 2 Groq classifier with anti-injection wrapper + circuit breaker, log-only (PR-M2 of [ADR-0013](docs/architecture/adr/0013-ai-moderation-pipeline.md)).** Third of eight PRs landing the AI moderation pipeline. New `ModerationStage2` service is a self-contained Groq client (does NOT depend on ChatBotLLMService so a noisy MovieBot neighbour can't starve moderation calls — separate `MODERATION_GROQ_API_KEY` env override is available for ops who want quota isolation). The system prompt pins three category definitions (hate_speech / threat / sexual), explicit use-vs-mention rules, AAVE / reclamation / song-lyric / educational-use guidance, and the **anti-prompt-injection contract**: every chunk of untrusted user content is wrapped in `<transcript untrusted="true">…</transcript>` and the system prompt explicitly tells the model that content inside is data, never instructions. JSON schema is enforced via `response_format: { type: 'json_object' }`; the sanitizer drops unknown categories, clamps `risk_level` to 0..3, and slices `explanation` to 2000 chars. Per-call `AbortController` timeout (8s default). **Circuit breaker**: 5 consecutive failures (5xx, 429, network error, JSON parse failure, verdict-shape failure) open the breaker for 60s during which `classify()` returns `{degraded: true, reason: 'breaker_open'}` without making the API call. A single success resets the failure counter. `ModerationService` now consumes the Stage 2 instance via constructor injection, only calls `classify()` when Stage 1 returned matches (demand-gated, so we don't pay Groq for clean chunks), passes a 60-second per-streamer rolling context buffer alongside the Stage 1-tripping chunk, and persists the full Stage 2 verdict (`stage2_verdict_json`, `stage2_risk_level`, `stage2_categories_json`) into `moderation_events`. A Stage 2 verdict of `degraded` or `error` produces `final_decision = 'deferred_degraded'` — the system never auto-acts during a Groq outage; admin review picks up the slack (M3 wires the action arbiter, this PR stays log-only so even `risk_level=3` stays `admin_review`). **Sliding-overlap Stage 1 matching**: the matcher now runs over `previousChunkText + currentChunkText` so a multi-word threat phrase split across the 5s chunk boundary still lands intact in at least one window — defeats the cross-window-fragmentation attack from the red-team review. Per-streamer rolling buffers (`_contextBuffers`, `_previousChunkText`) are time-trimmed against `contextWindowMs` (default 60s) on every push so memory doesn't grow unbounded for long sessions; `stop()` clears them alongside the mutex chain. `server/index.js` instantiates `ModerationStage2` immediately before `ModerationService` and injects it via the constructor's `stage2` dep, falling back to `process.env.GROQ_API_KEY` when `MODERATION_GROQ_API_KEY` isn't set. **Tests**: 27 new tests across two files. `ModerationStage2.test.js` (~21 tests) covers prompt construction (system-prompt anti-injection contract present, user-prompt wrapping, surrounding-context ordering), happy-path verdict parsing with the exact `response_format` request shape, degraded paths (no API key, empty transcript, non-2xx status, malformed envelope, malformed verdict JSON, verdict missing required fields, unknown categories dropped by sanitizer), circuit-breaker lifecycle (opens at threshold, refuses calls without invoking fetch during the open window, closes after `cbOpenMs`, single success resets counter), and a **prompt-injection corpus** of 5 payloads (D1 classifier-override monologue, D2 tool-hijack, D3 fabricated medical exemption, closing-tag injection, role-juggling) asserting that every payload lands fully enclosed inside the untrusted delimiters when the user message is built. `ModerationService.test.js` adds Stage 2 integration tests: classify only called on Stage 1 hit, surrounding context plumbed through, verdict persisted to the row, degraded → `deferred_degraded`, error → `deferred_degraded`, Stage 2 not ready → row still written with `admin_review` (no Stage 2 call), and the sliding-overlap split-phrase test ("i would never kill" + "all jews i swear" → caught on the second chunk). 478/478 server tests pass (+27 new).
- **AI moderation pipeline — ModerationService scaffold + Stage 1 word filter, log-only (PR-M1 of [ADR-0013](docs/architecture/adr/0013-ai-moderation-pipeline.md)).** Second of eight PRs landing the AI moderation pipeline. New `ModerationService` (stateful, EventEmitter, registered as `app.locals.moderationService` and `global.moderationService` per the conventions) subscribes to `TranscriptionService`'s `transcription-chunk` event and runs each chunk through Stage 1: normalize (NFKC + lowercase + homoglyph fold for Cyrillic/Greek look-alikes + leetspeak fold for `0→o`, `1→i`, `3→e`, `4→a`, `5→s`, `7→t`, `8→b`, `@→a`, `$→s`, `!→i` + non-alphanumeric strip + 3+-repeat collapse + whitespace collapse), then match against an in-memory cache of all `enabled=1` rows from `moderation_terms`. Stage 1 is **recall-only**: every hit lands in a `moderation_events` row with `final_decision='admin_review'`, never `auto_ban` — Stage 2 (PR-M2, Groq classifier) and Stage 3 (PR-M3, OpenAI omni cross-check) are the gates that produce action verdicts. On any hit, `ModerationNotifier.eventCreated()` fans the new row to the `'admin'` Socket.IO room so the admin events tab (PR-M5) refreshes live; clean chunks write no row to avoid blowing up `moderation_events` with negative data. New `ModerationStage1.js` is the pure normalize/match module (no DB, no IO); new `ModerationNotifier.js` is the emit chokepoint mirroring the BuffNotifier/ViewerCountNotifier/StreamNotifier pattern with four methods (`eventCreated`, `actionTaken`, `streamerBanner`, `botOutputDropped`) and a `MODERATION_EVENT_DECISIONS` Set that mirrors the schema CHECK constraint. **Per-streamer mutex** serializes processing for the same streamer via a `Map<streamerId, Promise>` chain — chunks from different streamers run concurrently, chunks from the same streamer run FIFO so PR-M2's Stage-2-latency coalescing has a stable insertion point. The mutex chain is cleaned up lazily when the streamer's chain settles to avoid unbounded map growth on transient streamers. **`ModerationService.initialize()` is async**: applies `ai-moderation-schema.sql` (idempotent — same `CREATE TABLE IF NOT EXISTS` pattern WhitelistService uses), verifies the embedded seed's SHA-256 against the sibling `.sha256` file (**fails closed** by default — refuses to start on mismatch, matching the WhitelistService pattern; toggleable via `AI_MODERATION_FAIL_CLOSED=false` for dev), upserts the embedded seed into `moderation_terms` with `source='embedded'` (re-enables embedded rows that an admin disabled — the boot-wins policy means a permanently-off embedded term needs the seed file edited and the SHA-256 refreshed, not a runtime override that vanishes on restart), and loads the enabled-terms cache. **Wired into the bootstrap**: `ModerationNotifier` joins the other three notifiers in `server/bootstrap/services.js`'s factory (constructed with `io`, added to the services bag — expected-key count bumped 39 → 40 and pinned in the bootstrap test); `ModerationService` itself is instantiated inline in `server/index.js` immediately after the `WhitelistService` init block because its `initialize()` is async, with the same try/catch + env-gated require-or-degrade pattern (`AI_MODERATION_REQUIRE_SERVICE=true` re-throws the init error; default is to log and continue without enforcement, the safe default for M1). **No runtime enforcement** — no bans, no rotations, no blocklist writes, no MovieBot output gating. Those land in M3, M3, M3, and M4 respectively. The `AI_MODERATION_ENFORCE` flag M3 will introduce defaults to `false` so even on a host that's fully wired through M7, enforcement is opt-in until M6 flips the default. **Tests**: 59 new tests across three new files. `ModerationStage1.test.js` covers normalization invariants (NFKC fullwidth fold, Cyrillic/Greek homoglyph fold, leet substitutions, punctuation strip, 3+-repeat collapse with doubles preserved, whitespace collapse, combined attacks) plus matching semantics (word-boundary for single-word terms, substring for multi-word phrases, regex-special-char safety in needles, span reporting, multi-match ordering, integration tests proving leetspeak/fullwidth/homoglyph/punctuation-padded slurs survive normalization to match the embedded seed terms; New dep `better-sqlite3@^11` alongside (not replacing) `sqlite3@^5.1.7`. New `server/database/database-better.js` exports `createBetterSqlite3Adapter(dbPath, {tuneForLargeReads, stmtCacheLimit})` → `{db, runAsync, getAsync, allAsync, close, walActive, cacheStats}`. - **Atomic `addPoints` / `subtractPoints` in `AccountService` (PR 5.1, first Phase 5 PR — [ADR-0013](docs/architecture/adr/0013-atomic-sql-for-mutable-counters.md)).** Closes the lost-update race documented in `server/tests/services/AccountService.points-race.test.js` (the failing-test artifact shipped during Phase 2). Both methods were a read-compute-write loop: `SELECT points_balance` → JS arithmetic → `UPDATE user_stats SET points_balance = ?` with the absolute new value. Two concurrent callers (e.g. a buff purchase that subtracts and a chat-points award that adds in the same tick) both `SELECT`ed the same stale balance, both computed against it, both `UPDATE`d — second write overwrote first. Reproducible deterministically with a 20-way `Promise.all` of `addPoints(userId, 5)` against the existing microtask-interleaving DB mock (the race test asserted `toBeLessThan(N*amount)` while the bug was live). New impl: `addPoints` issues a single `UPDATE user_stats SET points_balance = points_balance + ?, updated_at = CURRENT_TIMESTAMP WHERE user_id = ? RETURNING points_balance` via `getAsync`; if no row is returned (target row didn't exist) it falls through to `INSERT INTO user_stats (user_id, points_balance) VALUES (?, ?)` with the increment as the initial balance. `subtractPoints` issues a guarded `UPDATE … SET points_balance = points_balance - ? … WHERE user_id = ? AND points_balance >= ? RETURNING points_balance` so the insufficient-balance check is part of the same atomic statement — two concurrent debits cannot both pass a JS-side `if (currentBalance < amount)` guard and then both succeed. On guard miss, a follow-up `SELECT points_balance` populates the error string (`Insufficient points balance. Has: X, Needs: Y`); that SELECT can race with concurrent mutations and report a slightly-stale balance in the *error message* — the *decision* (refuse the debit) is correct, only the narrative drifts. **Test contract flip**: `AccountService.points-race.test.js` now asserts `toBe(expectedIfAtomic)` for both add and subtract instead of the pre-fix `toBeLessThan` / `toBeGreaterThan` gates. The microtask-interleaving DB mock is updated to understand the new SQL shapes: the guarded subtract regex (longest pattern, with `AND points_balance >= ?`) matches before the plain add regex; both branches mutate the in-memory `userStats` row atomically per call and return `{points_balance}` for the RETURNING semantic. The third test (sequential `addPoints` control) is unchanged and still asserts the exact arithmetic answer — proves the math, not just the locking. **Callers unchanged**: the ~14 `addPoints` / `subtractPoints` consumers in `server/routes/internal.js`, `server/services/ShopService.js`, and `server/services/TimeTrackingService.js` all consume the return value (the post-write balance) which is byte-equivalent to the old impl; no signature change. **`runAsync` → `getAsync` shift**: `RETURNING` rows flow through `db.get`, not `db.run` (which discards rows), so the SQL is now issued via the existing `getAsync` helper. This bleeds slightly into PR 5.2's better-sqlite3 adapter design — the adapter must route `getAsync` through a `prepare().get()` path that understands `RETURNING`. Cross-referenced in ADR-0013's "Implementation notes". **`UNIQUE(user_id)` on `user_stats` NOT added in this PR**: a narrow second-tick race remains where two concurrent first-time `addPoints` callers both see no existing row and both `INSERT`. Production data shows zero duplicates today on the 2026-05-27 snapshot (`SELECT user_id, COUNT(*) FROM user_stats GROUP BY user_id HAVING COUNT(*) > 1` returns empty), and realistic concurrency for a *first* award to a *new* user is essentially zero (gated behind signup + first chat / first viewing minute / first stream). A follow-up PR can add the constraint + a backfill-safe `ALTER TABLE` migration when broader schema cleanup lands. **No schema change, no new dependency**, behaviour-equivalent for every non-racy code path, race-free for every racy one. **One small behaviour addition**: the new `UPDATE` now also sets `updated_at = CURRENT_TIMESTAMP` on `user_stats` rows; the pre-PR `UPDATE` didn't, so `updated_at` could lag the actual last-mutation by an arbitrary amount. Strictly more truthful; a `grep` finds zero consumers today that filter or order by `user_stats.updated_at`, but flagged here so a future query author isn't surprised. SQLite engine: `node-sqlite3@5.1.7` links engine 3.44.2 (≥ 3.35 required for `RETURNING`, confirmed via `SELECT sqlite_version()`). 3/3 tests pass on `AccountService.points-race.test.js`. **Phase 5 plan**: PR 5.2 (better-sqlite3 adapter behind an env flag, preserves the existing `runAsync`/`getAsync`/`allAsync` shape across all ~356 callsites) lands next; PR 5.3 (repository-pattern rollout for the 3–5 most-touched DB-touching services) is conditional on what 5.1 + 5.2 leave behind.

Converted the noisiest startup sites: `server/index.js` env-check banner, `server/database/database.js` connect + PRAGMA logs, `server/database/applyPragmas.js` WAL-fallback warning, `server/routes/bug-reports.js` connection errors. The rest of the codebase still uses `console.*` and migrates opportunistically — this PR establishes the pattern without sweeping every file.
- **Fail-fast env validation at startup.** New `server/bootstrap/env.js` aggregates checks for the four secrets `.env.example` marks REQUIRED (`JWT_SECRET`, `SESSION_SECRET`, `TURN_SECRET`, `TURNSTILE_SECRET_KEY`) — both presence and a minimum-length floor where it applies — and throws a single error listing every missing/malformed var. Wired into `server/index.js` immediately after `dotenv.config()` and before any other `require()` that imports a file calling `requireEnv()`. Operators now see all env problems in one error instead of the previous "fix one, restart, see next miss, repeat" loop with `requireEnv()`. Existing `requireEnv()` callsites stay as a secondary safety net for any required var added to a route file but not yet added to `bootstrap/env.js`'s schema. `ADMIN_KEY` is deliberately not in the schema — it's documented as a legacy key without a length requirement; its existing `requireEnv` enforces presence.

### Removed
- **Legacy `user_stats.points` column dropped.** The migration to `points_balance` as the authoritative column (originally executed by `server/migrations/migrate-points-system.js`) has been complete for some time, and a grep across the codebase confirmed the `points` column was no longer read on any write or read path. Removed from the `CREATE TABLE` definition in `server/database/database.js`; the prior `ALTER TABLE user_stats ADD COLUMN points` migration block was replaced with an idempotent `ALTER TABLE user_stats DROP COLUMN points` (guarded by `"no such column"` for second-run safety). The `/api/auth/me` endpoint that serves `stats.points` to the client already overrode that field with `points_balance` (`routes/auth.js:285–288`), so the API contract is unchanged — `UserProfile.tsx` continues to display the authoritative balance. The standalone `migrate-points-system.js` historical script gained an at-start guard that detects the column-gone state and exits cleanly (preserved for forensic value but no longer re-runnable). `docs/architecture/data-model.md` updated to remove the legacy column from the documented schema.

### Changed
- **SQLite tuned for the 2 GB+ live DB.** New `server/database/applyPragmas.js` helper applies the project-wide PRAGMA set on every handle that opens `onestreamer.db`: `journal_mode=WAL` (with verification — silent fallback no longer leaves `synchronous=NORMAL` on a rollback journal, a power-loss corruption hazard), `synchronous=NORMAL` (conditional on WAL being active), `foreign_keys=ON`, `busy_timeout=5000`, plus large-read tunings (`temp_store=MEMORY`, `mmap_size=256 MB`, `cache_size=64 MB`) on the main handle. Before this change: live DB was on `delete` journal + `synchronous=FULL`, and the two auxiliary handles (`routes/bug-reports.js`, `services/URLStreamDatabaseService.js`) opened with no PRAGMAs at all — so writes from those handles were silently running with `foreign_keys=OFF` and `busy_timeout=0`, a pre-existing correctness regression that WAL would have made more visible (`SQLITE_BUSY` on collision). The pre-existing `sqlite3 ... ".backup"` in `docs/operations/backup-restore.md` is already WAL-safe; the inline comment there explicitly anticipated this. Takes effect on the next `pm2 restart onestreamer-server`.
- **Viewbot URL-stream encoder defaults retuned for lower CPU.** `ViewBotURLService` `adaptiveConfig.mode` default flipped from `'balanced'` to `'performance'` (maps to x264 `ultrafast` + 0.7× bitrate multiplier in `AdaptiveEncodingSettings`). The fixed-fallback path (which fires only on probe failure) was aligned: `superfast`/4000k/4500k/6000k → `ultrafast`/2000k/2500k/4000k. Adaptive remains the hot path. Observed: per-bot ffmpeg CPU dropped from ~90% to ~67% of a single core on a Kick HLS source at 720p30.
- **`onestreamer-client` PM2 entry removed** from `config/ecosystem.config.js`. The entry ran `react-scripts start` (CRA dev server) in "production" and was crash-looping (59 restarts in PM2). nginx already serves the built SPA from `/var/www/html` and never proxied to `127.0.0.1:3443`. Recovers ~1.2 GB RAM. Rollback: restore the entry and `pm2 start config/ecosystem.config.js --only onestreamer-client`.
- **CLAUDE.md** and **docs/architecture/overview.md** inline references to "LiveKit dormant" updated to point at [ADR-0008](docs/architecture/adr/0008-revive-livekit-for-url-streams-and-recording.md). The ADR landed but these inline mentions were missed.

### Added
- **`LIVEKIT_INGRESS_BYPASS_TRANSCODING` env var (default off)** in `ViewBotLiveKitService.createIngress`. When `true`, calls `createIngress` with `bypassTranscoding: true` and no `encodingOptions`, allowing LiveKit ingress to pass upstream H.264/AAC through unchanged (skips ~60% CPU of re-transcoding per active ingress). The team's prior "RTMP requires transcoding for WebRTC conversion, can't bypass" comment is preserved as context; flag exists so the claim can be re-validated on the current `livekit-server-sdk` version. **Requires device-QA (desktop Chrome + mobile Safari) before flipping in production** — mismatched SDP/keyframe interval can cause subscriber-side black streams.
- **`VIEWBOT_STREAM_COPY` env var (default off)** in `ViewBotURLService._createFFmpegRTMPProcess`. When `true` and the input is a direct URL (most IVS/Kick/Twitch HLS sources are H.264 + AAC already), uses `-c:v copy -c:a copy -bsf:v h264_mp4toannexb` instead of x264 re-encoding (~70% CPU off the viewbot ffmpeg). **Same QA requirement**; silent failure modes exist if a platform changes its encoder.

### Fixed
- **Admin Connections tab no longer hangs.** The `/admin/connections` handler in `server/index.js` tried to instantiate `new AccountService()`, but that class is never imported in the file — only the bootstrap-built `accountService` instance (destructured at module scope). The synchronous `ReferenceError` was thrown inside an async handler, which Express 4 leaves un-responded; clients hung until their own timeout. Removed the shadow; the outer-scope `accountService` is reused.
- **Admin Recording Review tab no longer hangs.** The `/admin/review/timeline` handler ran `SELECT * FROM recording_stream_segments` with no time filter (28,931 rows) and then did **one or two follow-up DB queries per segment** for display-name lookups. Response could exceed 20 MB and many seconds. Added a `?days=N` window filter (defaults to 30) and batched the per-segment lookups into a single `WHERE source_url IN (…)` + a single channel-ID `LIKE` per IVS channel + a single `streaming_logs` `WHERE streamer_id IN (…)`. Same data shape returned; timeline at `days=30` now responds in ~0.9 s with 3.3 MB.
- **URL-stream relay (Twitch/Kick) produces video again.** Operational fix; see ADR-0008 below. Verified end-to-end: FFmpeg → RTMP → `livekit-ingress` → LiveKit room → transcription / recording / MovieBot replies.
