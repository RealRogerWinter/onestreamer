### Added
- **URL-relay whitelist scaffolding (PR-W1, Phase 0 of ADR-0010).** New `WhitelistService` (`server/services/WhitelistService.js`) is the future single source of truth for the URL-relay content filter — three modes per platform (`off` / `blacklist` / `whitelist`) with optional fallback category + evergreen channel. Three new SQLite tables in `server/database/url-relay-whitelist-schema.sql`: `url_relay_filter_config` (one row per platform: mode + fallback fields), `url_relay_filter_entries` (allow/block streamer logins or category names, unique on `(platform, entry_type, value, list)`), `url_relay_filter_audit` (append-only log of mode changes, entry adds/removes, drift events, fallback engagements). Seed loader populates the tables on first init from `server/data/seeds/url-relay-whitelist.seed.json` — 35 Twitch streamers (Bob Ross 24/7 + Monstercat FM + Chillhop as evergreens) + 20 safe categories, 8 Kick streamers + 13 categories, with risk flags and notes per entry. `WhitelistService.checkAllowed({platform, login, currentGameName, isMature, ccls, hasMatureContent})` returns `{allowed, reason, gateThatBlocked}` and runs CCL/mature-flag gates (Twitch `SexualThemes`/`ViolentGraphic`/`DrugsIntoxication`, Kick `has_mature_content`) **regardless of mode** — those are platform-supplied signals, defense-in-depth against an admin allowlisting something whose labels change later. `filterCandidates(platform, candidates)` is the bulk-filter hook for rotation, `chooseFallback(platform)` returns the fallback configuration for the orchestrator, mutations (`setMode`/`addEntry`/`removeEntry`/`markReviewed`) emit `change` events and write audit rows with `before_json` / `after_json`. In-memory cache invalidates on mutation; stale-cache refresh is non-blocking so a slow disk doesn't block policy checks. Service is wired in `server/index.js` immediately after `SimpleViewBotRotation.setStreamService(streamService)` (the existing branch-shared hoist) and exposed on `app.locals.whitelistService` + `global.whitelistService` per the existing pattern. **Phase 0 only — no caller consults the service yet.** `ViewBotURLService` gate, `TwitchRandomService`/`KickRandomService` candidate filter, and `WhitelistEnforcer` drift checker land in PR-W2–W4. If `URL_RELAY_REQUIRE_WHITELIST_SERVICE=true` is set (planned for Phase 5/PR-W6), initialize failure is fatal; without that flag it's a logged warning so partial rollout stays safe. 30 new tests in `server/tests/WhitelistService.test.js` (in-memory SQLite, no network) cover: schema apply + idempotent re-init; CCL gates fire in `off` mode (`is_mature`, `SexualThemes`, `has_mature_content`); blacklist streamer + category blocks; whitelist allow by streamer OR by category; case-insensitive login match; CCL precedence over whitelist allow; `filterCandidates` end-to-end; `chooseFallback` returns evergreen list; every mutation writes an audit row with the right shape; cache invalidates within a single `checkAllowed` cycle after `setMode`; validation rejects unknown platforms/modes/empty value; seed loads the expected `bobross` evergreen and `mustafa_go` Kick allow entry. 288/288 overall (+30 new). **Schema parser fix**: the inherited split-on-`;` SQL parser (mirrored from `URLStreamDatabaseService._initializeSchema`) was filtering out CREATE statements that had leading `-- comment` lines because `s.startsWith('--')` was true after split-trim; the new implementation strips line comments per-line before checking emptiness, which is the actual robust behavior every other schema file in this repo was relying on by accident. **Documentation**: ADR-0010 (`docs/architecture/adr/0010-url-relay-whitelist-mode.md`, accepted), implementation plan (`docs/architecture/plans/url-relay-whitelist-mode.md`, 6 phases mapped to 6 PRs), operations runbook (`docs/operations/runbooks/url-relay-whitelist.md` — add/remove entries, fallback diagnostics, 90-day re-review query, force-stop incident response). **Known follow-up scope**: the parallel hardcoded `blockedCategories: ['ASMR', 'Pools, Hot Tubs, and Beaches']` on `RandomStreamRotationService.js:106` (and the duplicated filter inside `TwitchRandomService.findRandomStreamer:253` + `KickRandomService.findRandomStreamer:170`) are LEFT IN PLACE this PR; PR-W3 (Phase 2) deletes them after the rotation service starts consulting `WhitelistService` instead.

- **`ViewerCountNotifier` collapses 13 `viewer-count-update` emit sites into one chokepoint (PR 3.2).** Second PR of Phase 3, following the same chokepoint pattern as PR 3.1's `StreamNotifier` but with smaller blast radius and a slightly different shape. The 13 pre-existing emit sites were all **literally identical**: `io.emit('viewer-count-update', sessionService.getUniqueViewerCount())`. No payload variations, no `extras`, no per-site fields — the only thing varying was the file/line. New [`server/services/ViewerCountNotifier.js`](server/services/ViewerCountNotifier.js) owns both the emit AND the count derivation; callers now just write `viewerCountNotifier.broadcast()` and the notifier reads `sessionService.getUniqueViewerCount()` internally. The headline benefit beyond deduplication: the previous pattern forced every callsite to remember the right helper — `sessionService.getUniqueViewerCount()` (canonical unique-IP count) vs. `streamService.getViewerCount()` (raw socket-id count, multi-tab counts twice) was always one autocomplete away from a wrong-helper bug; with the chokepoint owning the lookup, that wrong choice is no longer available at the callsite. **Construction**: `new ViewerCountNotifier(io, sessionService)` — both args required (constructor throws otherwise). Wired in [`server/bootstrap/services.js`](server/bootstrap/services.js) immediately after `streamNotifier`; `expectedKeys` bumps 36 → 37. **Threading**: `StreamHandler` (3 sites at `:163`, `:709`, `:803`), `MediaSoupHandler` (2 sites at `:355`, `:385`), and `server/index.js` (8 sites — `:1945`, `:1979`, `:2023`, `:2059`, `:2087`, `:2793`, `:4908`, `:5127`) all get the notifier via their existing `deps` bag / module-scope destructure; no new construction order constraints because all 13 sites already had access to both `io` and `sessionService`. **Behaviour preservation**: wire format byte-equivalent — same event name `'viewer-count-update'`, same payload (single integer = `sessionService.getUniqueViewerCount()`'s return value), same per-emit freshness contract (the test pins "re-reads count on each broadcast" so a future `setInterval`-driven cache that staleness-shifts emits won't slip in silently). Zero is passed through deliberately (a "skip if zero" optimisation would create stuck-counter bugs on full-viewer-departure). **Tests** (`server/tests/services/ViewerCountNotifier.test.js` — 6 new tests): ctor throws on missing io OR missing sessionService; broadcast() reads the helper once and emits with the value; back-to-back broadcasts re-read (no caching); zero is emitted, not skipped. Bootstrap test (`server/tests/bootstrap/services.test.js`) updated to mock + identity-pin `ViewerCountNotifier` ctor `(io, sessionService)` and bump `expectedKeys` to 37. 476/476 (was 469; +6 new + 1 bootstrap identity). **Scope deferred**: a future PR could expose an event-driven model where `sessionService` emits "viewer-count-changed" and the notifier subscribes — that would eliminate the remaining "remember to call broadcast()" coupling at the 13 callsites. Postponed because it's a bigger contract change (sessionService would need an EventEmitter base) and the chokepoint pattern is enough to unblock the next steps. **Next**: PR 3.3 (BuffService.update() chokepoint for streamer-buffs-update / inventory-updated / buff-error, 28 sites split between BuffDebuffService and inline route handlers).

Counted precisely: **17 emit sites across 8 files** — 16 `io.emit('stream-ended', …)` calls plus 1 `socket.broadcast.emit('stream-ended', …)` for the takeover variant in `StreamHandler.js:414` (which deliberately EXCLUDES the new streamer's own socket so its UI doesn't process its own stream as "ended"). ### Fixed
- **Random-stream rotation settings persist when changed from the admin UI.** `RandomStreamRotationService.updateSettings()` (`server/services/RandomStreamRotationService.js:1952`) merged new values into `this.settings` and mirrored them onto the inner Twitch/Kick services, but never called `_saveState()` — only `start()` (:712) and `stop()` (:917) wrote `server/data/random-rotation-state.json`. So any change made via `PUT /api/random-stream/settings` (the admin panel) was lost on the next `pm2 restart`: the constructor's `_loadState()` read the file (which still held whatever value was current the last time someone toggled the rotation) and silently reverted the change. Concrete repro: set `minViewers=499` in the UI, observe `👁️ Viewer range set: 499 – 999999` in the log, restart, find `minViewers` back at the previously-persisted value (and the UI now also shows that older value). Fix is a single `this._saveState()` call at the end of `updateSettings()` so every UI write is durable. `_saveState()` already wraps in try/catch, so a transient disk error is logged but doesn't break the in-memory update path. New tests in `server/tests/services/RandomStreamRotationService.test.js` (2 tests): `updateSettings({minViewers: 499, maxViewers: 123456})` writes both fields into the on-disk state file; a fresh service instance loaded after a separate `updateSettings({minViewers: 777})` picks up `777` from disk without `start()` being called.
- **Recording cleanup no longer races B2 upload (PR 2.6).** Closes the documented hazard in `docs/architecture/background-work.md` "Notable hazards". `ContinuousRecordingService.cleanupOldRecordings()` (the body at :1378, fired by the 60-second interval at :1368) used to unconditionally `fs.rmSync` any `session_<timestamp>` directory older than `retentionMinutes` (= 10 min stock — `bootstrap/services.js:210`). `RecordingUploadScheduler.localBufferHours` waits 2 h after `end_time` before queueing the upload (`RecordingUploadScheduler.js:15`), and the upload's success path is the only thing that sets `b2_file_id` (:178). If the cleanup ever matched a session whose upload hadn't yet completed, it would nuke the local files 110 min before the upload fired; the upload then fails at :147 with "Local recording not found", retries every 30 min against a missing path, and the row sits at `status = 'completed'` / `b2_file_id = NULL` forever — recording permanently lost. **Live-exposure note**: the current production session-id generator at :818 produces `recording_<YYYY-MM-DD>` (date-based; one session per day), which does NOT match the cleanup regex `/session_(\d+)/` or the `.startsWith('session_')` guard — so in today's running system the cleanup loop is effectively a no-op and the race hasn't been actively destroying recordings. The fix is preemptive: the existing `extractSessionIdFromEgress` path at :940-942 hints that the per-session naming format can re-converge, and the loop-as-written is one config flip or one refactor away from the live failure mode. **Fix**: at the top of `cleanupOldRecordings()`, preload the set of session_ids that have NOT yet been uploaded (`SELECT session_id FROM recording_sessions WHERE b2_file_id IS NULL` — same predicate the upload scheduler queries at `RecordingUploadScheduler.js:52`), and skip those directories in the loop. Covers both `status='completed'` (pending) and `status='processing'` (in flight) since both have `b2_file_id IS NULL` until the upload sets it. The single-file `room_<ts>.mp4` / `.json` branch is left untouched: grep confirms no current code path produces matching filenames (egress writes `playlist_<ts>.m3u8` and `seg_<ts>_<idx>.ts` inside session directories, never `room_*` at the output-dir root), so the legacy cleanup is harmless dead code with no `recording_sessions` row to gate against. Cleanup is now async (the DB lookup needs await); the `setInterval` callback at :1368 fires-and-forgets the promise, which is the correct shape for a periodic cleanup. **Fail-closed**: if the DB query itself throws (e.g. `SQLITE_BUSY` under heavy write contention), the cleanup aborts the tick without deleting anything and the next tick retries — better to delay cleanup than to race-delete an unconfirmed upload's source. 7 new tests in `server/tests/services/ContinuousRecordingService.cleanup-race.test.js`: (1) headline — pending session_id stays on disk, (2) uploaded session_id is deleted, (3) mixed-batch discrimination (4 sessions, 2 of each), (4) retention cutoff still respected for uploaded sessions, (5) currently-active session is skipped, (6) orphan directory (no DB row) falls through to deletion (intentional — alternative would be a slow-motion disk-fill bug), (7) fail-closed on DB error. DB and `fs` operations are mocked at the module-require level (same pattern as PR 2.1's `AccountService.points-race.test.js`); no fake timers because the production race is between two scheduled intervals, not microtask interleaving inside one method. 206/206 (was 199; +7). **Known follow-up scope**: (a) the parallel `RecordingCleanupScheduler.js` (B2/DB row deletion) has the same gap and needs its own gate against `b2_file_name IS NULL`; (b) the upload scheduler's `loadPendingUploads` (:50-54) queries only `status='completed'` — a server crash mid-upload leaves rows orphaned at `status='processing' AND b2_file_id IS NULL`, which the cleanup now also (correctly) skips, but with no escape hatch they sit on disk forever; (c) currently no metric/alarm when `skippedPendingUpload > 0` for many consecutive ticks (a real B2 outage would silently fill disk). These each warrant a separate PR.
- **Kick random-stream picker honours `minViewers` again.** `server/services/kick-api-helper.py` now requests `https://kick.com/stream/livestreams/<lang>?…&sort=desc` — without `sort=desc`, that endpoint returns `viewer_count: 0` for every entry, so the helper was feeding the picker a list of fake-zero streams. `KickRandomService.findRandomStreamer` had been papering over this with a `if (minViewers > 1 && viewers < minViewers) return false;` guard that turned the viewer-count filter off at the default `minViewers: 1`; the moment an operator raised the threshold (e.g. 1 → 499) every Kick stream was filtered out and rotation recovery logged `⚠️ No suitable Kick streams found after filtering`. With real counts now coming back, the filter collapses to the obvious `viewers < minViewers || viewers > maxViewers` form. With `sort=desc`, page N covers a contiguous viewer-count band (page 1: ~6k–46k, page 3: ~3k–4.7k, page 5: ~1.6k–2k, page 10: ~700–830, page 20: ~330–345), so the existing random-page-1-through-5 sweep still surfaces a healthy range without hitting the long tail. Test coverage in `server/tests/services/KickRandomService.test.js` pins: streams below `minViewers` are dropped when `minViewers > 1`; zero-viewer streams are dropped at `minViewers === 1`; streams above `maxViewers` are dropped; an all-too-low list returns `null`.
- **URL-stream relay for Twitch no longer fails when popular streams are picked.** `server/services/URLStreamExtractorService.js` `getStreamURL()` now resolves Twitch URLs to a direct HLS m3u8 via `yt-dlp -g` and returns `{pipeMode: false, isHLS: true}`, so FFmpeg consumes the playlist directly (same code path Kick already uses). Previously Twitch fell into the streamlink pipe, and streamlink 6.6.2's `plugins/twitch.py` hardcodes `headless=False` when acquiring its client-integrity token — which means it launches Chromium with a real window and polls `http://127.0.0.1:<port>/json/version`. On this headless host there is no `DISPLAY`, Chromium dies with "Missing X server or $DISPLAY", and streamlink exits with `error: Unable to open URL: …/json/version`, which bubbled up as repeated `FFmpeg exited early with code 183` in the rotation logs. The integrity-token requirement is enforced more aggressively on high-viewer Twitch channels, so raising `minViewers` (e.g. 1 → 499) made the failure mode constant instead of intermittent. Test coverage in `server/tests/services/URLStreamExtractorService.test.js` pins: yt-dlp is invoked for Twitch and the result is `tool: 'yt-dlp', pipeMode: false, isHLS: true`; YouTube/Kick/Facebook continue to use streamlink pipe mode; direct `.m3u8` URLs still bypass both extractors unchanged.

### Added
- **Client adopts `streamGeneration` + deletion of the 10-second takeover lock (PR 2.5b — pair to PR 2.5).** Server-side seam from PR 2.5 (`StreamService.streamGeneration` counter, bumped on every `setStreamer` / `clearStreamer`, included in every `stream-status` payload via `getStreamStatus()`) is now consumed on the client, and the wall-clock-window lock that PR 2.5 was structurally replacing is deleted. New `client/src/hooks/useStreamGenerationGuard.ts` (custom hook) tracks the highest-seen counter on a `useRef` (initial −1 so the server's first emit at 0 is accepted), and exposes a memoized `acceptStreamStatus(incoming)` predicate that returns `false` (and the caller drops the payload) when `incoming < lastSeen`. Equal counters are accepted (same generation, fresh viewer-count / display-name resolution). Payloads with no `streamGeneration` field — older servers, or partial emit sites that don't yet thread the counter — are accepted unconditionally (back-compat: can't reason about ordering without the counter, safer to apply than to silently drop). The guard also subscribes to the socket's `connect` event and resets its high-water mark to −1 there: a server restart rewinds `StreamService.streamGeneration` to 0, and without the reset the client would lock itself out of every subsequent `stream-status` forever. `App.tsx`'s `stream-status` listener is rewritten: the 10-second `takeoverTargetRef` / `takeoverTimestampRef` lock (lines 272-323 pre-PR), including both the "ignore stale during transition" branch and the "clear lock on match" branch, is replaced by a single `if (!acceptStreamStatus(status.streamGeneration)) return;` guard at the top of the handler. The post-guard body preserves the existing `streamerDisplayName` and rotation-timer fall-through merge so display continuity through transitions is unchanged. `useStreamState` is cleaned up to match: the `takeoverTargetRef` and `takeoverTimestampRef` declarations (`useStreamState.ts:161-162`), interface entries (138-139), all set sites (`stream-ended` takeover branch :316-318, `stream-takeover` :669-670), all clear sites (`stream-started` :281-283, post-transition `setTimeout` :698), and return entries (:745-746) are deleted; the hook's header comment is updated to reflect the new ordering source. `StreamStatus` gains an optional `streamGeneration?: number` field for type safety. The `GameStreamService` symmetry deferred in PR 2.5 is also closed: `streamService` is threaded into the ctor as a 4th positional arg (`server/bootstrap/services.js:236`), and both `startGameStream` and `stopGameStream` emits call `streamService.bumpStreamGeneration()` (new method on `StreamService` — explicit bump for callers that emit stream-status outside the `setStreamer`/`clearStreamer` path; returns the new counter) and include it in their custom payload via spread. Server: 199/199 (was 195). Client hooks suite: 32/32 (was 23). The pre-existing client test failures in `App.test.tsx`, `MediasoupClient.test.ts`, `WebRTCViewer.test.tsx`, etc. exist on `main` and are not introduced by this PR. **Scope considered but not done in this PR**: the `stream-started` / `stream-ended` / `stream-takeover` payloads in `useStreamState.ts` also drive `setStreamStatus`, and they don't currently carry a `streamGeneration` field — so they sit outside the guard. The headline ordering hazard is the `stream-status` channel (the canonical "current state" message), which is fixed; threading the counter onto the notification-style events is a follow-up if a real-world out-of-order arrival on those channels is observed.
- **`StreamService.streamGeneration` monotonic counter on stream-status emits (server-side seam for PR 2.5).** Per the WebRTC red-team finding, the takeoverTargetRef 10-second lock at `client/src/App.tsx:272-323` is hiding cross-channel ordering between `stream-status` events: the lock papers over stale stream-status events arriving after a takeover. The structurally correct fix is a monotonic generation counter threaded through every stream-status emit so the client can drop out-of-order arrivals by counter. This PR adds the server-side half: `StreamService.streamGeneration` (init 0), incremented in `setStreamer()` and `clearStreamer()` (every stream-identity change), exposed via `getStreamGeneration()` and included in `getStreamStatus()` return. Since the three `stream-status` emit sites in `server/sockets/StreamHandler.js` (lines 104, 518, 628) all build their payload from `streamService.getStreamStatus()` and the `enrichStreamStatus` helper does `{...status}`, the new field rides through automatically with no emit-site changes. Viewer add/remove deliberately does NOT bump the counter — only stream-identity changes do — so the client's drop-by-counter check fires only on stream-state messages, not on viewer-count-only updates that arrived with the same stream context. The factory wiring is unchanged (StreamService takes no deps). 5 new tests in `server/tests/StreamService.test.js` (17/17, up from 12); existing shape `toEqual` test updated to include the new field. 195/195 overall (was 190). **Scope deferred to PR 2.5b**: the client-side drop-by-counter logic at `App.tsx:272-323` + the matching site in `useStreamState.ts` and the deletion of `takeoverTargetRef` / `takeoverTimestampRef` (declarations at `useStreamState.ts:161-162`, return at :745, all set/clear sites at :281-283, :316, :669, :698, plus App.tsx consumer destructure at :189 and dep array at :424). **Also deferred**: `GameStreamService` emits at `services/game/GameStreamService.js:52,93` build their own payload and don't currently include the counter — its game-mode stream uses a constant streamerId (`GAME_STREAM_ID`) so ordering pressure is lower, but PR 2.5b should add the counter for symmetry (which requires threading streamService into the GameStreamService ctor; bootstrap test update needed).
- **Bootstrap fail-fast guard for the service-factory DAG.** Third PR of Phase 2. The existing 33 tests in `server/tests/bootstrap/services.test.js` each pin one service's ctor args by identity — catches *wrong* wiring. The iteration walks every entry in the returned `services` (and `bag`) object, skips entries without `_args` and `null` ViewBot branch slots, then reports a list of `serviceName#argIndex` offenders. New `server/tests/services/AccountService.points-race.test.js` proves the long-standing read-compute-write defect in `AccountService.addPoints` / `subtractPoints`: each method does `await getUserStats` → JS arithmetic → `UPDATE user_stats SET points_balance = ?` with the absolute new value, so two concurrent callers both read the same balance, both compute, both write — second write overwrites the first. A control test using sequential awaits asserts `toBe(100)` to prove the bug is the race itself, not the arithmetic. `subtractPoints` gets the same treatment. DB layer is mocked with an in-memory map + `Promise.resolve()` microtask yields between read and write; that's enough to expose the race in Node's single-threaded model and faithfully mirrors what real sqlite3 (which yields on I/O) does today. ### Changed
- **SQLite tuned for the 2 GB+ live DB.** New `server/database/applyPragmas.js` helper applies the project-wide PRAGMA set on every handle that opens `onestreamer.db`: `journal_mode=WAL` (with verification — silent fallback no longer leaves `synchronous=NORMAL` on a rollback journal, a power-loss corruption hazard), `synchronous=NORMAL` (conditional on WAL being active), `foreign_keys=ON`, `busy_timeout=5000`, plus large-read tunings (`temp_store=MEMORY`, `mmap_size=256 MB`, `cache_size=64 MB`) on the main handle.
