### Added
- **`LifecycleManager` registry for deferred one-shot work (PR 4.2, second Phase 4 PR — [ADR-0011](docs/architecture/adr/0011-lifecycle-manager.md)).** Closes the hazard documented in [`docs/architecture/background-work.md`](docs/architecture/background-work.md) "Notable hazards / `setTimeout` chains in `server/index.js` startup": at the start of Phase 4 there were **9 scheduler-candidate `setTimeout` callsites** (7 in `server/index.js`, 2 inside `server/sockets/DisconnectHandler.js` after PR 4.1) where the timer handle was discarded immediately, so `clearTimeout()` was impossible and a SIGTERM landing inside any of those grace windows fired the callback against half-torn-down state — sometimes a swallowed exception, sometimes a wedged child process, occasionally a zombie rotation cycle. New [`server/services/LifecycleManager.js`](server/services/LifecycleManager.js) is the registry: `schedule(name, fn, delayMs)` returns a handle and tracks it in a `Set`; `cancel(handle)` proactively drops one; `async stop()` clears every pending handle and refuses further `schedule()` calls. Errors from `fn` are caught + logged with the task name and original `delayMs` — symmetry with the existing inline pattern where every `setTimeout(async () => {...})` site already absorbed its own exceptions, plus the structured-log line now gives production telemetry an attribution path. **Two windows, not one** (review-feedback fix on PR 4.2): `stop()` closes both the *pre-fire* race (clearTimeout on every pending handle) AND the *post-fire-mid-await* race (tracks every in-flight fn promise in `this.inFlight` and `Promise.allSettled`s them before resolving). The first draft only handled the pre-fire window — the `async stop()` qualifier was misleading because there was nothing to await, and an in-flight fn would have run free against torn-down services. The shutdown loop's per-service 5-second `Promise.race` deadline still bounds the total wait. **`delayMs` validation** (review-feedback fix): non-numeric / NaN / negative / Infinity `delayMs` warns + returns null, mirroring the existing `name` and `fn` validators. `setTimeout(fn, NaN)` would have fired immediately with no attribution. **Wiring**: constructed in [`server/bootstrap/services.js`](server/bootstrap/services.js)'s `createServices` factory with no deps (`expectedKeys` bumps 38 → 39); appended **last** to the `stoppables` array so the shutdown loop's reverse-iteration drains it **first** — any pending deferred work scheduled against the other services is cancelled before those services tear down. Threaded into `server/index.js` via the existing module-scope destructure block alongside the three notifiers, and threaded into `DisconnectHandler.js` via the existing per-connection deps bag. **Relocated callsites** (7): `server/index.js:4077` (post-stream rotation, 5 s); `:4969` (random-rotation autostart, MediaSoup branch, 5 s); `:5068` (random-rotation autostart, LiveKit branch, 5 s); `:5259` (viewBotRotation start, 10 s); `:5667` (AccountDeletionScheduler start, 5 s); `DisconnectHandler.js` (viewbot-rotation restart after real-user disconnect, 3 s); `DisconnectHandler.js` (real-streamer status validate, 1 s). Each gets a unique name (`'post-stream-rotation'`, `'random-rotation-autostart-mediasoup'`, `'random-rotation-autostart-livekit'`, `'viewbot-rotation-start'`, `'account-deletion-scheduler-start'`, `'viewbot-rotation-restart-after-disconnect'`, `'real-streamer-status-validate'`) so the structured-log attribution is real. **Deleted callsites** (2 — dev-debug code that shipped to production unguarded): `server/index.js:5150` was a 5-second `global.io.emit('test-event', {test: true})` whose only purpose was to sanity-check `global.io` was set; the two preceding `typeof` log lines already cover that diagnostic. `:5706` iterated every authenticated session at boot and logged `getStreamerDisplayName(socketId) = "..."` for each — useful when the lookup was being added, dead chatter now and never gated behind `NODE_ENV`. Both deletions are behaviour changes (a motivated dev relying on either loses their affordance), but unguarded debug code is more noise than signal; if either is wanted back it should land behind `process.env.NODE_ENV !== 'production'`. **Untouched** (3 — different shapes, not scheduler candidates): `:5749` `setTimeout` inside a `Promise.race` deadline already correctly captures + clears its handle in the loop's `finally`; `:5920` `await new Promise(resolve => setTimeout(resolve, 500))` is a sleep, not a deferred callback; `:5705` is commented out. **Adjacent fix** (called out by ADR-0011's "Consequences"): the `'account-deletion-scheduler-start'` task body now also pushes the scheduler onto `stoppables` after `.start()`, closing a pre-existing gap where the scheduler's `setInterval` (15-day grace check, per background-work.md "Owned" table) was leaking on SIGTERM because the scheduler was constructed inside a setTimeout and so the bootstrap factory couldn't have registered it. This is a behaviour change adjacent to the headline relocation, but it's directly in scope because the deferred construction is exactly the failure mode this PR is closing. **Tests**: new `server/tests/services/LifecycleManager.test.js` (14 tests) covers schedule + cancel + stop + validation + error swallowing + edge cases (cancel-after-fire returns false, cancel-foreign-handle returns false, schedule-after-stop returns null). Bootstrap test (`server/tests/bootstrap/services.test.js`) updated: mock + identity-pin `LifecycleManager` ctor `()`; `expectedKeys` bumped to 39; 2 new tests (no-arg ctor, lifecycleManager is the last entry in `stoppables`). 65/65 pass on the bootstrap + LifecycleManager + AdminHandler suites. **Scope deferred**: PR 4.3 — full `startServer()` decomposition so Phase 5 (better-sqlite3 + atomic points updates) can start without the surrounding orchestration noise. Also requires an ADR. `background-work.md`'s "Notable hazards / `setTimeout` chains" entry is rewritten to record resolution + pointer to ADR-0011.

### Changed
- **Sweep the last six inline `socket.on(...)` listeners out of `server/index.js` (PR 4.1, first Phase 4 PR).** Per [`docs/architecture/socket-events-actual.md`](docs/architecture/socket-events-actual.md)'s "Listener concentration" table, `server/index.js` held 6 inline socket listeners as of the start of Phase 4 — the only file in the `services/` + `sockets/` tree still registering events outside an extracted handler. This PR completes the extraction. Two listeners (`request-stream`, `request-test-stream`) move into the existing [`server/sockets/StreamHandler.js`](server/sockets/StreamHandler.js) because they are the viewer-side companions to `stream-offer` / `stream-answer` / `stop-streaming`. Three listeners (`drawing-path-complete`, `drawing-path-start`, `drawing-path-update`) move into a new [`server/sockets/DrawingHandler.js`](server/sockets/DrawingHandler.js) — pure `socket.broadcast.emit` relays with no service touch, so the module takes no `deps` bag and its signature is `(io, socket)` rather than the `(io, socket, deps)` shape the other `register*Handler` modules use (review-feedback fix: an unused `_deps` parameter is dead weight). The sixth listener (`disconnect`) moves into a new [`server/sockets/DisconnectHandler.js`](server/sockets/DisconnectHandler.js) — the heaviest of the three because it orchestrates per-socket teardown across mediasoup, sessions, time tracking, ViewBot Plain RTP cleanup, takeover cooldowns, stream-ended chokepoint broadcast, and viewer-count fanout. **Behaviour preservation**: behaviour-equivalent (whitespace re-flowed when the indented inline body moved to top-level inside `module.exports = function`, so it isn't *byte*-equivalent — review feedback corrected the original CHANGELOG draft). Same control flow, same logging, same error swallowing on the transport-close `try/catch`. The 16-key `deps` bag for `DisconnectHandler` threads `mediasoupService`, `sessionService`, `timeTrackingService`, `notifiedStreamers`, `viewbotSocketIds`, `cleanupViewbotUsername`, `plainTransportService`, `streamService`, `takeoverService`, `streamingLogsService`, `streamNotifier`, `notifyViewersStreamEnded`, `viewerCountNotifier`, `SimpleViewBotRotation`, plus `getViewbotService` / `getViewBotClientService` as lazy getters (both are late-init, reassigned by `startServer` after MediaSoup setup — the getter pattern is the same one `StreamHandler` and `ViewBotHandler` use). The getter result is captured once per disconnect at the top of the handler so a mid-handler re-init can't half-apply across branches. **Registration order preserved**: `DisconnectHandler` registers its `disconnect` listener at the same point in the `io.on('connection', ...)` block where the inline one used to sit — after `registerGameHandler` (which also registers a scoped `disconnect`) and before `registerEffectHandler`. Same firing order as before. **Two nested `setTimeout`s in the disconnect body** (3 s viewbot-rotation restart after a real-user disconnect; 1 s real-streamer-status validation) are deliberately preserved in-place rather than relocated — PR 4.2 (the LifecycleManager work) takes them as part of the broader scheduler-registry effort, alongside the four boot-autostart `setTimeout`s in `startServer()` and the rotation-restart `setTimeout` after stream end. Inline TODO comments on both sites name PR 4.2 so a reviewer doesn't repropose the same relocation. **`server/index.js` shrinks by ~196 lines** (6198 → 6002). The file is now free of inline `socket.on(...)` calls; future drift back into inline-handler territory will be caught by the existing `grep` recipe at the bottom of `socket-events-actual.md`. **No new tests**: following the existing handler-extraction pattern. `AdminHandler` was the pilot and got a full integration-test file (`server/tests/sockets/AdminHandler.test.js`); subsequent extractions (BuffHandler, EffectHandler, GameHandler, MediaSoupHandler, StreamHandler, ViewBotHandler) all moved listeners and did not ship their own per-handler integration tests — the existing 49 bootstrap + AdminHandler tests still pass. This PR matches that precedent: the relocations are mechanical, the destination handlers are wired through the same `register*Handler(io, socket, deps)` shape that the other 7 handlers already use, and the bootstrap fail-fast guard (PR 2.3 / #99) continues to catch missing deps at the service-construction layer. **No ADR**: the brief flagged this as the smallest of the three Phase 4 PRs and "probably doesn't qualify" — confirmed, since the extraction continues an established pattern rather than making a structural choice. **Scope deferred**: (a) PR 4.2 — LifecycleManager for the 7 real scheduler-candidate `setTimeout` sites (4 boot autostarts at `server/index.js:5158/5257/5448/5856`; 3 event-reactive at `:4075/4899/4939` — the latter two are now inside `DisconnectHandler`). The two debug-only sites (`:5339` test emit, `:5895` `getStreamerDisplayName` smoke) are deletion candidates rather than scheduler candidates. The two non-scheduler shapes at `:5940` (Promise.race deadline with `clearTimeout`) and `:6111` (sleep) stay untouched. (b) PR 4.3 — full `startServer()` decomposition so Phase 5 (better-sqlite3 + atomic points updates) can start without the surrounding orchestration noise. Requires an ADR.

- **URL-relay whitelist drift enforcer (PR-W4, Phase 3 of ADR-0010).** Closes the last of three URL-relay surfaces — and the one a static allow/block list can't catch on its own: a whitelisted streamer switching their own category mid-broadcast to something not on the policy. New `WhitelistEnforcer` service runs a `setInterval` poll (default 60s) inside the main server process. Each tick: gets the active URL relay via `viewBotURLService.getActiveURLStream()`; if non-Twitch/non-Kick, skip; otherwise re-fetch a fresh snapshot via new `TwitchRandomService.getCurrentStreamSnapshot(login)` (`/helix/streams?user_login=` for game + is_mature, then `/helix/channels?broadcaster_id=` for CCL — the second is wrapped in a try/catch so a Helix degradation doesn't abort the check) or `KickRandomService.getCurrentStreamSnapshot(login)` (single `getChannelInfo` via the kick-api-helper Python subprocess); pass the snapshot to `WhitelistService.isStillAllowed`. On a non-allow decision, the enforcer calls `viewBotURLService.stopURLStream(urlId)` (the existing rotation failure handler turns this into a rotation), emits a new `whitelist-drift-stop` socket event (`{urlId, login, platform, reason, gateThatBlocked, currentGameName}`) so the admin UI in PR-W5 can surface what happened, and writes a `drift_block` audit row via `WhitelistService.logAudit`. **Platform-API degradation tolerance**: one snapshot fetch failure is silently skipped (`snapshot_failed_first`); subsequent failures inside a 3-minute window are still skipped (`snapshot_failed_recent`) — we keep last-known-good state and don't stop the stream on transient flakes. Beyond 3 minutes of failures we err on the safe side and stop (`platform_degraded_extended`), on the assumption that an extended outage shouldn't keep a possibly-out-of-policy relay running. Snapshot success clears the failure timer for that urlId. **Wiring**: instantiated in `server/index.js` on both MediaSoup and LiveKit branches immediately after `RandomStreamRotationService` is set up (the enforcer borrows the same `twitchService` + `kickService` instances the rotation service owns — no duplicate construction). Guarded by `if (whitelistService)` and `start()` is called immediately. The timer is `.unref()`'d so it doesn't keep the process alive on shutdown. **Login extraction**: `_loginFromActive(active)` delegates to `viewBotURLService.extractorService.extractIdentifier(url)` when available (PR-W2's helper pattern), with a naive regex fallback if not. **Interval source**: per the schema, `url_relay_filter_config.drift_check_seconds` exists (default 60), but `chooseFallback` doesn't currently surface it. Phase 4 (admin UI) will read this from the DB explicitly; for now the default constant is in `WhitelistEnforcer.DEFAULT_DRIFT_CHECK_SECONDS`. **Why a separate service rather than inside `WhitelistService`**: WhitelistService is pure policy (in-memory cache + DB writes). WhitelistEnforcer orchestrates I/O — platform polls, stream stops, socket emits. Keeping them separate lets us unit-test policy without mocking the world. **Tests**: 11 new tests in `server/tests/services/WhitelistEnforcer.test.js` covering: `start()` with missing deps no-ops (won't crash on a half-init host); `start()/stop()` lifecycle; tick with no active stream skips; YouTube relay skips (platform_not_gated); whitelisted streamer still in policy → no-op + no emit; streamer drifted to non-whitelisted category → stops + audits + emits with full payload shape (Should-not-leak-control-fields assertion); single snapshot failure → skipped; failure inside 3-min window → still skipped; failure beyond 3-min window → stop on the safe side; recovery after failure clears the timer; Kick path uses `kickService.getCurrentStreamSnapshot` and does NOT call `twitchService`. The `setInterval` lifecycle itself is trivial and exercised by start/stop. 360/360 overall (+11 enforcer tests + the two new `getCurrentStreamSnapshot` helpers are exercised indirectly through the enforcer's mocked stubs; integration with the live Helix API is operator-smoke-tested). **What's deferred to PR-W5 (Phase 4)**: admin UI surfacing the `whitelist-drift-stop` event, reading `drift_check_seconds` from DB, and a stats endpoint for "how many drift stops in the last 24h." **What's deferred to PR-W6 (Phase 5)**: setting `URL_RELAY_REQUIRE_WHITELIST_SERVICE=true` in prod, finalizing defaults.
