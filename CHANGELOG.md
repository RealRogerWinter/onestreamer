### Changed
- **`console.*` → namespaced pino sweep, top 20 services (PR 12.2, Phase 12 — applies [ADR-0020](docs/architecture/adr/0020-namespaced-logging-with-pino.md)).** Second PR of Phase 12. Applies the convention codified in PR 12.1 (#152 / ADR-0020) to the 20 highest-density service files identified in the inventory. **Quantitative**: in-scope `console.*` count drops from **3750 → 1664** (-2086, -56 %). ### Added
- **Namespaced-logging convention codified + `console.*` callsite inventory (PR 12.1, Phase 12 — [ADR-0020](docs/architecture/adr/0020-namespaced-logging-with-pino.md)).** First PR of Phase 12 (observability sweep). Docs-only — zero code changes. Codifies the convention the sweep PRs (12.2, 12.3) will land against and inventories the surface so the sweep has a concrete target list. **Why now**: at Phase 11 close the codebase has **3750 `console.*` callsites across 128 server-side files** (verified by `grep -rE "console\.(log|info|warn|error|debug)" server/{services,routes,bootstrap,middleware,database} --include="*.js"`), against a [`server/bootstrap/logger.js`](server/bootstrap/logger.js) module that exposes a single shared pino logger but is imported by only **six** server files — three of which use it cleanly (`database-better.js`, `applyPragmas.js`, `transaction.js`) and three of which are hybrid (`server/index.js` is the canonical "logger imported but unused" pattern at 1 logger call vs 368 console calls). Two operational consequences: (a) no filterable log structure — `process.stdout.write(util.format(...) + '\n')` mixes 96 services' output with no level / service-tag / structured timestamps to index against; (b) no correlation across layers — when a socket emit goes wrong, the originating HTTP route and the chokepoint notifier (Phase 3 — `StreamNotifier`, `ViewerCountNotifier`, `BuffNotifier`) have no shared identifier and "what happened to this user's request?" requires reading timestamps and guessing. Phase 12 closes both. PR 12.1 codifies; PR 12.2 sweeps the top 20 services; PR 12.3 sweeps the tail and adds `_traceId` propagation via `AsyncLocalStorage`. **ADR-0020 decisions**: (1) one pino child logger per service / route file, namespaced by a `svc` binding (`require('../bootstrap/logger').child({ svc: 'ChatBotService' })`) — so every log line carries `"svc":"ChatBotService"` in the JSON payload and Loki / ELK / Datadog can filter natively. Child loggers are cheap (pino caches bindings); construct one at module scope per file. (2) Level conventions pinned: `fatal` (process about to die), `error` (handled error — request failed, service continues; always include `err` so pino serializes the stack), `warn` (suspicious state, degraded operation, retries), `info` (noteworthy lifecycle — service started, scheduled task ran, user-visible state transition), `debug` (per-request / per-event detail for production troubleshooting; default-on in dev, default-off in prod), `trace` (very-noisy interior loops; production-off). The boundary that matters most operationally is **info ↔ debug** — info lines should read as a story of what the system did at a high level. (3) What stays on `console.*`: migration scripts under `scripts/migrations/` (CLI tools, no shared logger context, 526 callsites — explicitly out-of-scope per Phase 12 brief's success criterion), other `scripts/` (ops, deploy, setup), and `server/bootstrap/logger.js` itself (circular if it logs through itself). Everything else migrates. (4) `_traceId` propagation (target shape; lands in PR 12.3): every chokepoint notifier emit carries `_traceId`; trace ID generated at HTTP-route entry via express middleware and propagated through the service call chain via `AsyncLocalStorage` so every `logger.X(...)` call inside the request scope picks up `traceId` as a binding automatically. A production debugging session collapses to `journalctl -u onestreamer --since '15 min ago' | grep '"traceId":"abc123"'` and gets every line from the HTTP route through the service through the socket emit. The propagation helper lands in PR 12.3 as `server/bootstrap/trace-context.js` (~80 LoC); PR 12.2 prepares the way by sweeping services to use the namespaced logger — once services call `logger.X(...)`, the AsyncLocalStorage layer can inject `traceId` into the bindings transparently and every existing log line gets the field for free. **Inventory** (in [`docs/architecture/plans/console-callsite-inventory.md`](docs/architecture/plans/console-callsite-inventory.md)): top 20 service files by `console.*` density cover **2127 calls** = 57 % of all in-scope callsites; PR 12.2's touch list is those 20 files (~500 lines edited). Tail bucket: ~1623 calls across ~108 files (average ~15 calls/file) — PR 12.3's touch list. The headline-density-top-three is the Phase-11 ViewBot pair plus VisualFx: `server/services/viewbot/ViewBotInstance.js` (369, the Phase 11 extraction), `server/services/ViewBotClientService.js` (197, post-extraction orchestrator), `server/services/VisualFxService.js` (163). Per-directory breakdown: services 3256/96, routes 409/25, database 44/2, bootstrap 23/3, middleware 18/2. The methodology block is in the inventory so PR 12.2 / PR 12.3 can re-run the grep against the merge-tip-of-main for a fresh snapshot before scoping. **Strict sequencing constraint preserved**: Phase 12 lands AFTER Phases 7, 8, 9, 10, 11 — the logging sweep touches every service file, and if it landed before, every Phase 7–11 diff would be contaminated by logger churn. PR 12.1 ships now that all of Phases 7–11 are merged on main (PRs #134, #142–149, #150, #151). **Bonus docs cleanup**: the ADR README index ([`docs/architecture/adr/README.md`](docs/architecture/adr/README.md)) at the Phase 11 close was missing the ADR-0019 row that PR 11.1 (#150) should have added. PR 12.1 adds both the missing 0019 row and the new 0020 row in one shot — same table, defensible scope creep given this PR is docs-only. **Diff**: `docs/architecture/adr/0020-namespaced-logging-with-pino.md` +110 (new); `docs/architecture/plans/console-callsite-inventory.md` +95 (new); `docs/architecture/adr/README.md` -1 / +3 (two new ADR rows); `CHANGELOG.md` +1 entry. Zero changes under `server/`, `client/`, `chat-service/`, `scripts/`. **Tests**: not run for this PR (docs-only — no code surface to break). PR 12.2's touch will re-establish the suite baseline. **Risk**: LOW (per Phase 12 brief — docs-only). **No reviewer subagent pass** — docs-only PR, the convention is straightforward and matches the existing logger.js docstring's stated migration intent ("new code (and any of the noisiest existing sites a refactor touches) has a structured target"). **Live smoke**: N/A (no behaviour change). **Phase 12 status**: 1/3 PRs done. - **Recording-cleanup vs upload-retry data-loss race (PR 8.4, Phase 8).** Fourth and final functional PR of Phase 8 (reliability hardening). Closes the database half of the cleanup-vs-upload race documented in [`docs/architecture/background-work.md`](docs/architecture/background-work.md) under "Notable hazards → Recording cleanup races recording upload". PR 2.6 already closed the **filesystem** half (`ContinuousRecordingService.cleanupOldRecordings` skips directories whose `recording_sessions.b2_file_id IS NULL`); this PR closes the matching gap in `RecordingCleanupScheduler.runCleanup` which deletes the `recording_sessions` **database row** (and the B2 file if present) after retention. The test replicates the new SQL filter in the DB mock so the bind-shape matches production exactly; the assertions exercise the real `runCleanup` and `getStatus` paths. Full suite: **845/846 under `--runInBand`** (same pre-existing `database-better.test.js` flake from the Phase 6 handoff; reproducible on main without this PR; 9/9 new tests pass). **Risk**: MEDIUM (data-loss avoidance — the fix is small and the test covers each branch of the new SQL guard). **No ADR** — single-statement SQL-clause addition without an architectural decision; the design is fully described in this entry and in the in-file comments. **Honesty**: the existing `RecordingCleanupScheduler.deleteSession` still deletes the B2 file via `b2Storage.deleteFile(session.b2_file_name)` before deleting the DB row — that order is preserved, and a deletion that gets past the new guard already has `b2_file_name` set (or null, in which case the B2 call is a no-op for un-uploaded safety-valve deletions). - **ViewBot child-process force-reap on shutdown (PR 8.3, Phase 8 — extends [ADR-0011](docs/architecture/adr/0011-lifecycle-manager.md)).** Third PR of Phase 8 (reliability hardening). Closes the orphan-process leak documented in [`docs/operations/runbooks/viewbot-fleet-misbehaving.md`](docs/operations/runbooks/viewbot-fleet-misbehaving.md): a ViewBot `gst-launch-1.0` spawned by `SimpleViewBotRotation` whose lifecycle is interrupted (server SIGTERM mid-rotation, lifecycle exception during teardown) was being orphaned because the legacy `global.processManager.trackProcess(...)` registration paths were no-ops (the global was never assigned, so the typeof-guarded calls silently did nothing). **Mechanism**: extends the existing `ProcessManager` singleton with `reapAll({ graceMs = 2000 })` / `stop()` methods, wired into the shutdown loop's `stoppables` array immediately before `lifecycleManager`. The reaper iterates every tracked PID: skip if already dead at entry; otherwise SIGTERM via process-group kill (negative-PID), poll every 100 ms until grace expires, then SIGKILL anything still alive. Both signal phases are gated by a comm-snapshot PID-reuse defense — `/proc/<pid>/comm` is captured at register time and re-read before each signal; if it drifted, the kernel recycled the PID into an unrelated process and the signal is skipped (better to leak a real orphan than to SIGTERM/SIGKILL an innocent sshd). The signal-send path tries process-group kill first (cheaper, brings down gst-launch + children together) and falls back to direct-PID kill if the process wasn't a group leader. **Production-path coverage**: the registry call in `SimpleViewBotRotation.startMediaSoupBot` now actually populates via `processManager.registerProcess(bot.id, 'gstreamer', this.gstreamerProcess.pid)`; the gst-launch spawn gains `detached: true` (non-Windows) so the child becomes its own process-group leader — matches the precedent in `ViewBotClientService` and ensures the reaper's negative-PID kill targets the gst-launch group, not Node's own PGID. A natural-exit handler (`gstreamerProcess.once('exit', ...)`) deregisters via `processManager.onBotStopped(bot.id)` so the registry doesn't accumulate ghost PIDs that could collide with PID-reused unrelated processes at shutdown time. **Stoppables ordering**: `processManager` sits immediately above `lifecycleManager` in the factory array — reverse-iteration of `stoppables` runs `lifecycleManager.stop()` FIRST (cancel deferred work), then `processManager.stop()` (reap straggler PIDs). - **`withTransaction` helper (PR 7.1, Phase 7 — [ADR-0015](docs/architecture/adr/0015-transaction-shape-for-multi-statement.md)).** First foundational PR of the Phase 7 money-flow refactor. Adds `withTransaction(fn, { busyTimeoutMs })` to `server/database/database.js`. Wraps a body fn in `BEGIN IMMEDIATE … COMMIT` (or `ROLLBACK` on throw), serialized at the JS layer via a process-wide Promise-chain mutex so two concurrent scopes cannot interleave statements on the shared connection. Body fn receives a `tx = { runAsync, getAsync, allAsync }` proxy that routes through the same wrappers — repo methods accept `tx` as a dep override and work unchanged whether invoked inside or outside a transaction. Works under **both backends** (`USE_BETTER_SQLITE3=true` and `=false`) using `BEGIN`/`COMMIT` as SQL statements via the existing `runAsync` wrapper — deliberately NOT better-sqlite3's `db.transaction(fn)` because that requires a sync body and the extracted repos return promises. The optional `busyTimeoutMs` parameter applies a per-scope `PRAGMA busy_timeout` (restored to the 5000 ms default on exit, success OR failure) for callers that want to fail fast under writer-lock contention; defaults to whatever the connection currently has set (the 5000 ms default established by `applyPragmas.js` / `database-better.js`). **Red-team finding closed**: under sqlite3, `await runAsync('BEGIN IMMEDIATE')` does NOT by itself serialize subsequent `await runAsync(...)` against the same handle — between the BEGIN's resolution and the next statement's submission, any unrelated caller can submit a statement that runs on the same connection inside our open tx. The mutex closes the in-process hole; the IMMEDIATE lock closes the cross-handle hole. The remaining hazard (a fire-and-forget `runAsync` from a timer/socket handler during an open scope) is documented as a sharp edge in the helper's doc-comment and the ADR — discipline-enforced rather than code-enforced, because funneling every write in the codebase through `withTransaction` would be massively over-scoped for this PR. Implemented as a factory (`createWithTransaction({ runAsync, getAsync, allAsync })`) so tests can instantiate isolated instances against in-memory DBs; production `database.js` instantiates once at module-load time with the post-`USE_BETTER_SQLITE3`-swap wrappers and exports a singleton. **Tests**: 10 contract `it()` blocks in `server/tests/database/transaction.test.js` cover happy-path commit, body-throw rollback, statement-error rollback (UNIQUE constraint violation), post-rollback recovery, two-concurrent-scopes serialization order (asserts the labels arrive A,A,B,B not A,B,A,B), error-scope-doesn't-block-next-scope (mutex releases on throw), busy_timeout PRAGMA application + restore on success + restore on failure. Each runs under both backends via `describe.each([{flag:'true'},{flag:'false'}])`, so 20 distinct test invocations total. **Side cleanup — ADR-0013 number collision resolved**: `0013-atomic-sql-for-mutable-counters.md` (PR 5.1's atomic-points ADR) and `0013-ai-moderation-pipeline.md` (the AI moderation ADR) both shipped on main with the same number from parallel branches. A control test using sequential awaits asserts `toBe(100)` to prove the bug is the race itself, not the arithmetic. `subtractPoints` gets the same treatment. DB layer is mocked with an in-memory map + `Promise.resolve()` microtask yields between read and write; that's enough to expose the race in Node's single-threaded model and faithfully mirrors what real sqlite3 (which yields on I/O) does today. All 183 existing tests still pass.
- **`BotEventBus` decouples ChatBot ↔ MovieBot.** Third step of Phase 1. New `server/services/BotEventBus.js` (a thin `EventEmitter` wrapper with a `stop()` lifecycle hook) is constructed once in the factory and injected into both `ChatBotService` (constructor arg `{ botEventBus, getMoviePromptTemplate }`) and `MovieBotService` (5th positional arg). Forward-declared with `let movieBotService` so the closure captures by reference; the construction-time circular dep is broken without losing the runtime config read. `MovieBot → ChatBot` direction is unchanged: that direction is request/response (`generateMovieComment`, `getMovieBotEnabledBots`, `llmService` config) and contorting it through a bus would be ceremony. Also fixed in the same PR: PR 1.2 changed `createServices`/`createViewBotServices` to return `{ services, stoppables }` but didn't update `server/tests/bootstrap/services.test.js`; 20+ test callsites are migrated to `const { services: s } = createServices(...)`, the deleted `setMovieBotService` test is replaced with bus-wiring tests that assert both services hold the same bus instance and that `emit('chat-message')` reaches subscribers, and `services.test.js` `expectedKeys` array adds `'botEventBus'` (35 keys, was 34). - **Graceful-shutdown iteration over a `stoppables` registry.** Second step of Phase 1: `server/bootstrap/services.js`'s `createServices()` and `createViewBotServices()` now return `{ services, stoppables }`. `stoppables` is a list of services that actually expose `stop()` today (the four bot/effect services with leaked anonymous intervals — `visualFxService`, `chatBotService`, `movieBotService`, `transcriptionService`, plus `viewBotLiveKitService` — are deferred to later PRs that add real `stop()` rather than no-op fallbacks). `server/index.js` aggregates them: core stoppables from the factory + livekit (when on the LiveKit branch) + viewbot stoppables + two more inline-constructed services (`viewBotManager`, `portMonitor`). livekit is pushed *before* the viewbot stoppables so reverse-iteration stops viewbot-consumers before the underlying livekit client. The graceful-shutdown work is extracted into a single `shutdown(signal)` function that **both** `SIGINT` and `SIGTERM` call directly — the previous `process.emit('SIGINT')` from SIGTERM was fire-and-forget and never awaited the async handler, which made systemd / Docker shutdowns silently skip the cleanup work. Each `svc.stop()` runs as an async IIFE (catches sync throws) against a per-iteration cancelable timeout (no leaked unhandled rejections on fast successes). The existing inline cleanup block stays in place as belt-and-braces — a follow-up PR prunes entries that overlap with stoppables once the iterator is proven in production. `cleanupMediaProcesses()` (the `uncaughtException` emergency `pkill` path) is unchanged.

### Changed
- **SQLite tuned for the 2 GB+ live DB.** New `server/database/applyPragmas.js` helper applies the project-wide PRAGMA set on every handle that opens `onestreamer.db`: `journal_mode=WAL` (with verification — silent fallback no longer leaves `synchronous=NORMAL` on a rollback journal, a power-loss corruption hazard), `synchronous=NORMAL` (conditional on WAL being active), `foreign_keys=ON`, `busy_timeout=5000`, plus large-read tunings (`temp_store=MEMORY`, `mmap_size=256 MB`, `cache_size=64 MB`) on the main handle. Before this change: live DB was on `delete` journal + `synchronous=FULL`, and the two auxiliary handles (`routes/bug-reports.js`, `services/URLStreamDatabaseService.js`) opened with no PRAGMAs at all — so writes from those handles were silently running with `foreign_keys=OFF` and `busy_timeout=0`, a pre-existing correctness regression that WAL would have made more visible (`SQLITE_BUSY` on collision). The pre-existing `sqlite3 ... ".backup"` in `docs/operations/backup-restore.md` is already WAL-safe; the inline comment there explicitly anticipated this. Takes effect on the next `pm2 restart onestreamer-server`.
- **Viewbot URL-stream encoder defaults retuned for lower CPU.** `ViewBotURLService` `adaptiveConfig.mode` default flipped from `'balanced'` to `'performance'` (maps to x264 `ultrafast` + 0.7× bitrate multiplier in `AdaptiveEncodingSettings`). ### Added
- **`LIVEKIT_INGRESS_BYPASS_TRANSCODING` env var (default off)** in `ViewBotLiveKitService.createIngress`.
