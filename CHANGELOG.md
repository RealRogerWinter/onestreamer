## [Unreleased]

### Added
- **Client adopts `streamGeneration` + deletion of the 10-second takeover lock (PR 2.5b — pair to PR 2.5).** Server-side seam from PR 2.5 (`StreamService.streamGeneration` counter, bumped on every `setStreamer` / `clearStreamer`, included in every `stream-status` payload via `getStreamStatus()`) is now consumed on the client, and the wall-clock-window lock that PR 2.5 was structurally replacing is deleted. New `client/src/hooks/useStreamGenerationGuard.ts` (custom hook) tracks the highest-seen counter on a `useRef` (initial −1 so the server's first emit at 0 is accepted), and exposes a memoized `acceptStreamStatus(incoming)` predicate that returns `false` (and the caller drops the payload) when `incoming < lastSeen`. Equal counters are accepted (same generation, fresh viewer-count / display-name resolution). Payloads with no `streamGeneration` field — older servers, or partial emit sites that don't yet thread the counter — are accepted unconditionally (back-compat: can't reason about ordering without the counter, safer to apply than to silently drop). The guard also subscribes to the socket's `connect` event and resets its high-water mark to −1 there: a server restart rewinds `StreamService.streamGeneration` to 0, and without the reset the client would lock itself out of every subsequent `stream-status` forever. `App.tsx`'s `stream-status` listener is rewritten: the 10-second `takeoverTargetRef` / `takeoverTimestampRef` lock (lines 272-323 pre-PR), including both the "ignore stale during transition" branch and the "clear lock on match" branch, is replaced by a single `if (!acceptStreamStatus(status.streamGeneration)) return;` guard at the top of the handler. The post-guard body preserves the existing `streamerDisplayName` and rotation-timer fall-through merge so display continuity through transitions is unchanged. `useStreamState` is cleaned up to match: the `takeoverTargetRef` and `takeoverTimestampRef` declarations (`useStreamState.ts:161-162`), interface entries (138-139), all set sites (`stream-ended` takeover branch :316-318, `stream-takeover` :669-670), all clear sites (`stream-started` :281-283, post-transition `setTimeout` :698), and return entries (:745-746) are deleted; the hook's header comment is updated to reflect the new ordering source. `StreamStatus` gains an optional `streamGeneration?: number` field for type safety. The `GameStreamService` symmetry deferred in PR 2.5 is also closed: `streamService` is threaded into the ctor as a 4th positional arg (`server/bootstrap/services.js:236`), and both `startGameStream` and `stopGameStream` emits call `streamService.bumpStreamGeneration()` (new method on `StreamService` — explicit bump for callers that emit stream-status outside the `setStreamer`/`clearStreamer` path; returns the new counter) and include it in their custom payload via spread. The bootstrap test at `server/tests/bootstrap/services.test.js:410` is updated to assert `_args.length === 4` and `_args[3] === streamService`. **Tests**: 9 new client-side tests in `client/src/hooks/__tests__/useStreamGenerationGuard.test.tsx` covering: first-emit accept, equal-counter accept, strictly-higher advance, strictly-lower drop (the headline), back-compat for missing counter, socket-connect reset (server-restart case), connect-listener unbind on unmount (no leak), stable callback reference across renders (so the App.tsx `useEffect` dep doesn't churn), null-socket tolerance. 3 new server-side tests in `server/tests/services/GameStreamService.streamGeneration.test.js` covering: `startGameStream` bump + payload inclusion, `stopGameStream` bump + payload inclusion, omitted field when `streamService` is absent (back-compat). 1 new test in `server/tests/StreamService.test.js` for `bumpStreamGeneration()` itself. Server: 199/199 (was 195). Client hooks suite: 32/32 (was 23). The pre-existing client test failures in `App.test.tsx`, `MediasoupClient.test.ts`, `WebRTCViewer.test.tsx`, etc. exist on `main` and are not introduced by this PR. **Scope considered but not done in this PR**: the `stream-started` / `stream-ended` / `stream-takeover` payloads in `useStreamState.ts` also drive `setStreamStatus`, and they don't currently carry a `streamGeneration` field — so they sit outside the guard. The headline ordering hazard is the `stream-status` channel (the canonical "current state" message), which is fixed; threading the counter onto the notification-style events is a follow-up if a real-world out-of-order arrival on those channels is observed.
- **`StreamService.streamGeneration` monotonic counter on stream-status emits (server-side seam for PR 2.5).** Per the WebRTC red-team finding, the takeoverTargetRef 10-second lock at `client/src/App.tsx:272-323` is hiding cross-channel ordering between `stream-status` events: the lock papers over stale stream-status events arriving after a takeover. The structurally correct fix is a monotonic generation counter threaded through every stream-status emit so the client can drop out-of-order arrivals by counter. This PR adds the server-side half: `StreamService.streamGeneration` (init 0), incremented in `setStreamer()` and `clearStreamer()` (every stream-identity change), exposed via `getStreamGeneration()` and included in `getStreamStatus()` return. Since the three `stream-status` emit sites in `server/sockets/StreamHandler.js` (lines 104, 518, 628) all build their payload from `streamService.getStreamStatus()` and the `enrichStreamStatus` helper does `{...status}`, the new field rides through automatically with no emit-site changes. Viewer add/remove deliberately does NOT bump the counter — only stream-identity changes do — so the client's drop-by-counter check fires only on stream-state messages, not on viewer-count-only updates that arrived with the same stream context. The factory wiring is unchanged (StreamService takes no deps). 5 new tests in `server/tests/StreamService.test.js` (17/17, up from 12); existing shape `toEqual` test updated to include the new field. 195/195 overall (was 190). **Scope deferred to PR 2.5b**: the client-side drop-by-counter logic at `App.tsx:272-323` + the matching site in `useStreamState.ts` and the deletion of `takeoverTargetRef` / `takeoverTimestampRef` (declarations at `useStreamState.ts:161-162`, return at :745, all set/clear sites at :281-283, :316, :669, :698, plus App.tsx consumer destructure at :189 and dep array at :424). **Also deferred**: `GameStreamService` emits at `services/game/GameStreamService.js:52,93` build their own payload and don't currently include the counter — its game-mode stream uses a constant streamerId (`GAME_STREAM_ID`) so ordering pressure is lower, but PR 2.5b should add the counter for symmetry (which requires threading streamService into the GameStreamService ctor; bootstrap test update needed).
- **Bootstrap fail-fast guard for the service-factory DAG.** Third PR of Phase 2. The existing 33 tests in `server/tests/bootstrap/services.test.js` each pin one service's ctor args by identity — catches *wrong* wiring. Three new tests catch *missing* wiring: with the full `buildDeps()` bag, no positional ctor arg on any returned service should be `undefined`. The iteration walks every entry in the returned `services` (and `bag`) object, skips entries without `_args` and `null` ViewBot branch slots, then reports a list of `serviceName#argIndex` offenders. With the current factory, the offender list is `[]` — meaning future PRs that add a new service to `createServices` with `new X(deps.zservice)` but forget to add `zservice` to the destructure, or extend an existing service's ctor without threading the new dep, will fail with a useful name-and-index list instead of routes blowing up later when the undefined gets dereferenced at runtime. The check is `!== undefined` (not `!= null`) so deliberate `null` choices — `viewBotClientService` passing `null` as serverUrl for the env-var fallback, the unused LiveKit/MediaSoup branch leaving its `viewBot*Service` null on the bag, `livekitService=null` on the MediaSoup branch — pass through untouched. One guard for the main factory, two for `createViewBotServices` (MediaSoup branch + LiveKit branch, since each path hits a different `new` site). 36/36 in the suite (was 33), 190/190 overall.
- **TakeoverService concurrent-claim coverage.** Second PR of Phase 2. Four new tests under `server/tests/TakeoverService.test.js` close the gap between the existing 15 sequential-happy-path tests and the actual concurrent claim flow at `server/sockets/StreamHandler.js`. (1) `Promise.all` of two `canTakeOver()` calls with no `recordTakeover` between them returns `allowed=true` for both — pins down that TakeoverService is intentionally non-reserving so a future refactor doesn't silently start serializing here and mask the caller-layer gap PR 2.5 actually fixes with a monotonic `streamGeneration` counter on stream-status emits. (2) After `recordTakeover()`, three concurrent `canTakeOver()` calls all return `denied`/`global_cooldown`/`cooldownRemaining>0` — proves there is no stale-read window inside `canTakeOver`; will fail if anyone defers the in-memory `lastStreamStartTime` write past an `await`. (3) Per-identifier cooldown isolation: `setIpCooldown('alice', 'stream_taken_over')` does not block Bob's claim, but Alice herself sees `individual_cooldown` with `cooldownRemaining ≤ 60`. (4) Boundary test of the `(now - lastStreamStartTime) < globalCooldownMs` comparison: `lastStreamStartTime` 5 s ago → denied, 31 s ago → allowed. Tests target `globalCooldownSeconds`/`individualCooldownSeconds` directly (the live properties `canTakeOver` reads) rather than `setCooldownSeconds()` (the legacy field, which the existing "no-arg constructor leaves cooldownSeconds unset" test already pins). 19/19 in the suite, 187/187 overall.
- **Failing test documents the points-balance lost-update race.** First PR of Phase 2. New `server/tests/services/AccountService.points-race.test.js` proves the long-standing read-compute-write defect in `AccountService.addPoints` / `subtractPoints`: each method does `await getUserStats` → JS arithmetic → `UPDATE user_stats SET points_balance = ?` with the absolute new value, so two concurrent callers both read the same balance, both compute, both write — second write overwrites the first. The headline test fires 20 concurrent `addPoints(userId, 5, …)` calls and asserts the final balance is **strictly less than** the expected 100; it passes today (bug is live) and will fail when Phase 5 lands the atomic `UPDATE user_stats SET points_balance = points_balance + ? … RETURNING points_balance` fix — at which point the same diff flips the assertion to `toBe(100)`. A control test using sequential awaits asserts `toBe(100)` to prove the bug is the race itself, not the arithmetic. `subtractPoints` gets the same treatment. DB layer is mocked with an in-memory map + `Promise.resolve()` microtask yields between read and write; that's enough to expose the race in Node's single-threaded model and faithfully mirrors what real sqlite3 (which yields on I/O) does today. All 183 existing tests still pass.
- **`BotEventBus` decouples ChatBot ↔ MovieBot.** Third step of Phase 1. New `server/services/BotEventBus.js` (a thin `EventEmitter` wrapper with a `stop()` lifecycle hook) is constructed once in the factory and injected into both `ChatBotService` (constructor arg `{ botEventBus, getMoviePromptTemplate }`) and `MovieBotService` (5th positional arg). ChatBot now emits `chat-message` events on the bus instead of calling `movieBotService.addChatMessage(...)` directly; MovieBot subscribes in its existing `setupChatListener()` (handler captured on `this._onBusChatMessage` so a future `stop()` can `removeListener` cleanly). The post-construction setter `chatBotService.setMovieBotService(movieBotService)` was deleted from both the factory and the class — its only purpose was wiring the now-unnecessary cross-reference. The one other cross-service read (`createTemporaryBot` reads `movieBotService.config.moviePromptTemplate` — admin-editable, normally populated from the DB) is preserved via a factory-built closure (`getMoviePromptTemplate`) that lazily resolves `movieBotService.config.moviePromptTemplate` at call time. Forward-declared with `let movieBotService` so the closure captures by reference; the construction-time circular dep is broken without losing the runtime config read. `MovieBot → ChatBot` direction is unchanged: that direction is request/response (`generateMovieComment`, `getMovieBotEnabledBots`, `llmService` config) and contorting it through a bus would be ceremony. Also fixed in the same PR: PR 1.2 changed `createServices`/`createViewBotServices` to return `{ services, stoppables }` but didn't update `server/tests/bootstrap/services.test.js`; 20+ test callsites are migrated to `const { services: s } = createServices(...)`, the deleted `setMovieBotService` test is replaced with bus-wiring tests that assert both services hold the same bus instance and that `emit('chat-message')` reaches subscribers, and `services.test.js` `expectedKeys` array adds `'botEventBus'` (35 keys, was 34). All 33 tests pass.
- **Graceful-shutdown iteration over a `stoppables` registry.** Second step of Phase 1: `server/bootstrap/services.js`'s `createServices()` and `createViewBotServices()` now return `{ services, stoppables }`. `stoppables` is a list of services that actually expose `stop()` today (the four bot/effect services with leaked anonymous intervals — `visualFxService`, `chatBotService`, `movieBotService`, `transcriptionService`, plus `viewBotLiveKitService` — are deferred to later PRs that add real `stop()` rather than no-op fallbacks). `server/index.js` aggregates them: core stoppables from the factory + livekit (when on the LiveKit branch) + viewbot stoppables + two more inline-constructed services (`viewBotManager`, `portMonitor`). livekit is pushed *before* the viewbot stoppables so reverse-iteration stops viewbot-consumers before the underlying livekit client. The graceful-shutdown work is extracted into a single `shutdown(signal)` function that **both** `SIGINT` and `SIGTERM` call directly — the previous `process.emit('SIGINT')` from SIGTERM was fire-and-forget and never awaited the async handler, which made systemd / Docker shutdowns silently skip the cleanup work. Each `svc.stop()` runs as an async IIFE (catches sync throws) against a per-iteration cancelable timeout (no leaked unhandled rejections on fast successes). The existing inline cleanup block stays in place as belt-and-braces — a follow-up PR prunes entries that overlap with stoppables once the iterator is proven in production. `cleanupMediaProcesses()` (the `uncaughtException` emergency `pkill` path) is unchanged.
- **Uniform `async stop()` lifecycle method on 12 services.** First step of Phase 1 of the refactor (background-work lifecycle discipline). Each service in [`docs/architecture/background-work.md`](docs/architecture/background-work.md) "Owned" table now exposes an `async stop()` method. Sets up the next PR (factory returns a `stoppables` array; SIGTERM iterates) to dispatch through one method name across the whole service graph.
  - Ten services received a delegating alias to an existing teardown method (`ResourceMonitor`→`stopMonitoring`, `AudioOptimizationService`→`stopMonitoring`, `PortMonitorService`→`stopMonitoring`, `TimeTrackingService`→`stopPeriodicCleanup`, `LiveKitService`→`stopStreamerHealthCheck`, `BuffDebuffService`→`shutdown`, `CanvasFxService`→`shutdown`, `ContinuousRecordingService`→`shutdown`, `ViewBotManager`→`cleanup`, `ViewbotService`→`cleanup`, `ViewBotClientService`→`cleanup`).
  - Two services received fresh bodies: `StreamBotService.stop()` calls both `stopPeriodicMessages` and `stopAutoSummon`.
  - **Fixed two pre-existing leaks while wiring the entry points**: `BuffDebuffService.startCacheCleanup()` was storing the cache-cleanup interval handle anonymously, so `shutdown()` only cleared one of its two intervals — handle now stored as `this.cacheCleanupInterval` and cleared in `shutdown()`. `TimeTrackingService.stopPeriodicCleanup()` referenced `this.realTimeUpdateIntervals.entries()` but the attribute initialized in the constructor is named `this.updateIntervals` — that meant per-user interval cleanup was silently throwing on every call. Renamed to the actual attribute name.
  - **Scope caveat for PR 1.2**: three of the twelve services (`ViewBotManager`, `PortMonitorService`, `LiveKitService`) are constructed *outside* `bootstrap/services.js` today — `ViewBotManager` in `server/index.js:5461` and `WebRTCViewBotRotation.js:51`; `PortMonitorService` in `server/index.js:5498`; `LiveKitService` in `WebRTCAdapter.js` / `WebRTCAdapterV2.js`. PR 1.2's `stoppables` iterator needs to collect these alongside the factory-built bag.
  - **`ViewBotClientService.cleanup()` pre-existing footgun** (called transitively via the new `stop()` alias on SIGTERM): `pkill -f "puppeteer.*chrome"` is unscoped and would kill Puppeteer processes belonging to other projects on the same host. Not introduced by this PR; flagged for tightening as a Phase 1 follow-up (track launched browser PIDs in `activeBots`, kill those specifically).
- **Inventory artifacts under `docs/architecture/`.** Three new reference files captured at the start of the refactor:

  - [`background-work.md`](docs/architecture/background-work.md) — every `setInterval` / `setTimeout` / child-process site with lifecycle status (handle-on-instance vs leaked), grouped by severity. Crucial for the Phase 2 lifecycle work.

- **Structured logging (pino).** New `server/bootstrap/logger.js` exports a shared pino instance: JSON-to-stdout in production (consumable by log aggregators) and `pino-pretty` colored output in development. `LOG_LEVEL` env var overrides the level; defaults are `info` in production, `debug` in development. Converted the noisiest startup sites: `server/index.js` env-check banner, `server/database/database.js` connect + PRAGMA logs, `server/database/applyPragmas.js` WAL-fallback warning, `server/routes/bug-reports.js` connection errors. The rest of the codebase still uses `console.*` and migrates opportunistically — this PR establishes the pattern without sweeping every file.
- **Fail-fast env validation at startup.** New `server/bootstrap/env.js` aggregates checks for the four secrets `.env.example` marks REQUIRED (`JWT_SECRET`, `SESSION_SECRET`, `TURN_SECRET`, `TURNSTILE_SECRET_KEY`) — both presence and a minimum-length floor where it applies — and throws a single error listing every missing/malformed var. Wired into `server/index.js` immediately after `dotenv.config()` and before any other `require()` that imports a file calling `requireEnv()`. Operators now see all env problems in one error instead of the previous "fix one, restart, see next miss, repeat" loop with `requireEnv()`. Existing `requireEnv()` callsites stay as a secondary safety net for any required var added to a route file but not yet added to `bootstrap/env.js`'s schema. `ADMIN_KEY` is deliberately not in the schema — it's documented as a legacy key without a length requirement; its existing `requireEnv` enforces presence.

### Removed
- **Legacy `user_stats.points` column dropped.** The migration to `points_balance` as the authoritative column (originally executed by `server/migrations/migrate-points-system.js`) has been complete for some time, and a grep across the codebase confirmed the `points` column was no longer read on any write or read path. Removed from the `CREATE TABLE` definition in `server/database/database.js`; the prior `ALTER TABLE user_stats ADD COLUMN points` migration block was replaced with an idempotent `ALTER TABLE user_stats DROP COLUMN points` (guarded by `"no such column"` for second-run safety). The `/api/auth/me` endpoint that serves `stats.points` to the client already overrode that field with `points_balance` (`routes/auth.js:285–288`), so the API contract is unchanged — `UserProfile.tsx` continues to display the authoritative balance. The standalone `migrate-points-system.js` historical script gained an at-start guard that detects the column-gone state and exits cleanly (preserved for forensic value but no longer re-runnable). `docs/architecture/data-model.md` updated to remove the legacy column from the documented schema.

### Changed
- **SQLite tuned for the 2 GB+ live DB.** New `server/database/applyPragmas.js` helper applies the project-wide PRAGMA set on every handle that opens `onestreamer.db`: `journal_mode=WAL` (with verification — silent fallback no longer leaves `synchronous=NORMAL` on a rollback journal, a power-loss corruption hazard), `synchronous=NORMAL` (conditional on WAL being active), `foreign_keys=ON`, `busy_timeout=5000`, plus large-read tunings (`temp_store=MEMORY`, `mmap_size=256 MB`, `cache_size=64 MB`) on the main handle. Before this change: live DB was on `delete` journal + `synchronous=FULL`, and the two auxiliary handles (`routes/bug-reports.js`, `services/URLStreamDatabaseService.js`) opened with no PRAGMAs at all — so writes from those handles were silently running with `foreign_keys=OFF` and `busy_timeout=0`, a pre-existing correctness regression that WAL would have made more visible (`SQLITE_BUSY` on collision). The pre-existing `sqlite3 ... ".backup"` in `docs/operations/backup-restore.md` is already WAL-safe; the inline comment there explicitly anticipated this. Takes effect on the next `pm2 restart onestreamer-server`.
- **Viewbot URL-stream encoder defaults retuned for lower CPU.** `ViewBotURLService` `adaptiveConfig.mode` default flipped from `'balanced'` to `'performance'` (maps to x264 `ultrafast` + 0.7× bitrate multiplier in `AdaptiveEncodingSettings`). The fixed-fallback path (which fires only on probe failure) was aligned: `superfast`/4000k/4500k/6000k → `ultrafast`/2000k/2500k/4000k. Adaptive remains the hot path. Observed: per-bot ffmpeg CPU dropped from ~90% to ~67% of a single core on a Kick HLS source at 720p30.
- **`onestreamer-client` PM2 entry removed** from `config/ecosystem.config.js`. The entry ran `react-scripts start` (CRA dev server) in "production" and was crash-looping (59 restarts in PM2). nginx already serves the built SPA from `/var/www/html` and never proxied to `127.0.0.1:3443`. Recovers ~1.2 GB RAM. Rollback: restore the entry and `pm2 start config/ecosystem.config.js --only onestreamer-client`.
- **CLAUDE.md** and **docs/architecture/overview.md** inline references to "LiveKit dormant" updated to point at [ADR-0008](docs/architecture/adr/0008-revive-livekit-for-url-streams-and-recording.md). The ADR landed but these inline mentions were missed.

### Added
- **`LIVEKIT_INGRESS_BYPASS_TRANSCODING` env var (default off)** in `ViewBotLiveKitService.createIngress`. When `true`, calls `createIngress` with `bypassTranscoding: true` and no `encodingOptions`, allowing LiveKit ingress to pass upstream H.264/AAC through unchanged (skips ~60% CPU of re-transcoding per active ingress). The team's prior "RTMP requires transcoding for WebRTC conversion, can't bypass" comment is preserved as context; flag exists so the claim can be re-validated on the current `livekit-server-sdk` version. **Requires device-QA (desktop Chrome + mobile Safari) before flipping in production** — mismatched SDP/keyframe interval can cause subscriber-side black streams.
- **`VIEWBOT_STREAM_COPY` env var (default off)** in `ViewBotURLService._createFFmpegRTMPProcess`. When `true` and the input is a direct URL (most IVS/Kick/Twitch HLS sources are H.264 + AAC already), uses `-c:v copy -c:a copy -bsf:v h264_mp4toannexb` instead of x264 re-encoding (~70% CPU off the viewbot ffmpeg). **Same QA requirement**; silent failure modes exist if a platform changes its encoder.

### Fixed
- **Admin Connections tab no longer hangs.** The `/admin/connections` handler in `server/index.js` tried to instantiate `new AccountService()`, but that class is never imported in the file — only the bootstrap-built `accountService` instance (destructured at module scope). The synchronous `ReferenceError` was thrown inside an async handler, which Express 4 leaves un-responded; clients hung until their own timeout. Removed the shadow; the outer-scope `accountService` is reused.
- **Admin Recording Review tab no longer hangs.** The `/admin/review/timeline` handler ran `SELECT * FROM recording_stream_segments` with no time filter (28,931 rows) and then did **one or two follow-up DB queries per segment** for display-name lookups. Response could exceed 20 MB and many seconds. Added a `?days=N` window filter (defaults to 30) and batched the per-segment lookups into a single `WHERE source_url IN (…)` + a single channel-ID `LIKE` per IVS channel + a single `streaming_logs` `WHERE streamer_id IN (…)`. Same data shape returned; timeline at `days=30` now responds in ~0.9 s with 3.3 MB.
- **URL-stream relay (Twitch/Kick) produces video again.** Operational fix; see ADR-0008 below. Verified end-to-end: FFmpeg → RTMP → `livekit-ingress` → LiveKit room → transcription / recording / MovieBot replies.
